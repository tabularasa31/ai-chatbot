# Chat9 — описание продукта и план ручного тестирования

Документ для **тестировщика**: сначала коротко объясняется, что за продукт, затем — **что и в каком порядке проверять**. Тестирование API «вручную» (Postman и т.п.) помечено как необязательное; основной фокус — поведение через **веб-интерфейс** и **виджет**.

---

## Часть 1. Что это за продукт

### Название и идея

**Chat9** — веб-сервис для компаний и команд: загружаете документацию (PDF, Markdown, OpenAPI/Swagger и т.д.), система строит по ней **базу для ответов**, и пользователи могут **задавать вопросы на естественном языке** и получать ответы с опорой на ваши материалы (подход **RAG**: поиск по смыслу + генерация ответа языковой моделью).

### Кто пользователи

- **Владелец аккаунта** — регистрируется, подтверждает email, настраивает **рабочее пространство (client)** и ключ **OpenAI** (для эмбеддингов и чата на своих данных).
- **Конечные пользователи бота** — могут общаться через **встроенный виджет** на сайте клиента (без входа в Chat9).

### Изоляция данных (важно для тестов)

У каждого клиента свой **client** в системе: свои документы, свои чаты, свои ключи. Пользователь **не должен** видеть чужие документы и диалоги. Это нужно проверять отдельными сценариями с **двумя разными аккаунтами**.

### Основные части системы (где смотреть в интерфейсе)

| Область | Зачем |
|--------|--------|
| Лендинг / маркетинговая страница | Описание продукта, вход и регистрация |
| Регистрация, вход, восстановление пароля, подтверждение email | Доступ в продукт |
| Дашборд | API-ключ клиента, ключ OpenAI, код встраивания виджета, счётчики |
| Документы | Загрузка файлов, эмбеддинги, **проверка здоровья документа** (FI-032) |
| Чат (в приложении) | Проверка ответов по своим документам |
| Логи / сессии | История диалогов |
| Отзывы (👍/👎), разбор плохих ответов | Качество и обучение |
| Отладка (debug) | Технический просмотр режима поиска, фрагментов и **валидации ответа** (FI-034: `debug.validation` в API) |
| Админка метрик | Только для пользователей с флагом администратора |
| Страница виджета + скрипт встраивания | Публичный чат для сайта клиента |

### Внешние зависимости (для понимания сбоев)

- **OpenAI** — без валидного ключа у клиента не заработают эмбеддинги, чат и проверка здоровья документов.
- **Почта (Brevo)** — письма подтверждения и сброса пароля; если письма не приходят, проверяйте спам и настройки окружения на стенде.
- **Бэкенд и БД** — при недоступности API страницы покажут ошибки загрузки.

## Часть 2. Рекомендации для тестировщика

### Перед началом

1. Уточните у команды **URL тестового стенда** (или продакшена, если так решено) и **учётные данные тестового админа** (если нужна админка).
2. Заведите **два тестовых email** (или используйте плюс-алиасы вида `you+qa1@…`, `you+qa2@…`) для проверки изоляции.
3. Имейте под рукой **тестовый ключ OpenAI** (или отдельный проект/лимиты), если политика компании это разрешает.
4. Подготовьте **небольшие файлы**: `.md` с заголовками, маленький `.pdf` с текстом, при необходимости `.json` Swagger по шаблону из документации продукта.

### Как фиксировать баги

- **Шаги**: что делали по шагам (1, 2, 3…).
- **Ожидание** vs **факт**.
- Скриншот или короткое видео, **время**, **браузер**, **аккаунт** (без паролей).
- Если ошибка API — **код ответа** и текст ошибки (можно из DevTools → Network).

### Термины простым языком

- **Эмбеддинг** — числовое представление текста для поиска «по смыслу».
- **Чанк** — кусок документа, по которому строится поиск.
- **Rate limit** — ограничение числа запросов за время; при превышении может быть ошибка или задержка.

## Часть 3. План тестирования по модулям

Выполняйте блоки **сверху вниз**: сначала вход и дашборд, затем документы и чат, потом виджет и админку.

---

### Блок A. Дымовой тест (smoke) — 15–20 минут

Цель: убедиться, что продукт в целом живой.

- [ ] Открывается лендинг, нет критичных ошибок в консоли.
- [ ] Регистрация нового пользователя (или вход существующего).
- [ ] После входа открывается дашборд (или редирект в приложение).
- [ ] Страница «Документы» открывается.
- [ ] Выход из аккаунта и повторный вход.

Если smoke не проходит — остальное пока не тестируем, пишем блокер.

---

### Блок B. Аккаунт и безопасность входа

- [ ] **Регистрация**: валидация пароля/email (пустые поля, неверный формат email).
- [ ] **Подтверждение email**: переход по ссылке из письма; повторная регистрация с тем же email обрабатывается предсказуемо.
- [ ] **Вход**: неверный пароль — понятное сообщение; верный пароль — успех.
- [ ] **Восстановление пароля**: запрос письма → переход по ссылке → новый пароль → вход новым паролем.
- [ ] **Защищённые страницы**: без входа по прямой ссылке на `/dashboard` или разделы приложения — редирект на логин или отказ доступа.

---

### Блок C. Клиент (workspace), ключи, дашборд

- [ ] При первом входе создаётся клиент / отображаются **API key** и при необходимости **public_id** для виджета.
- [ ] **Сохранение OpenAI API key**: пустой, неверный формат (не `sk-…`), валидный — после сохранения интерфейс показывает, что ключ задан.
- [ ] **Имя клиента / обновление профиля** (если есть в UI) — сохраняется после перезагрузки страницы.
- [ ] **Код встраивания виджета** отображается и копируется (кнопка «копировать» не ломает страницу).

---

### Блок D. Документы

- [ ] Загрузка **разрешённых** типов: `.pdf`, `.md`, swagger `.json` / `.yaml` (как в подсказке на экране).
- [ ] **Отклонение** неподдерживаемого типа и **слишком большого** файла — понятное сообщение.
- [ ] Лимит числа документов (если в продукте заявлен, например максимум 20) — при достижении новая загрузка блокируется с ясным текстом.
- [ ] В списке документов отображаются имя, тип, статус (**processing / ready / error**).
- [ ] После загрузки при успешном сценарии документ переходит в **ready** и при необходимости автоматически запускаются **эмбеддинги** (как сейчас реализовано в UI).
- [ ] **Удаление** документа — исчезает из списка; повторная загрузка возможна.

---

### Блок E. Проверка здоровья документа (FI-032)

Детальный чеклист: **`docs/qa/FI-032-document-health-check.md`**.

Кратко здесь:

- [ ] После появления эмбеддингов обновляется индикатор здоровья (не «вечный Checking»).
- [ ] Цвета/подписи соответствуют диапазонам score (Good / Fair / Needs attention).
- [ ] Предупреждения видны при наведении и в раскрывающемся списке.
- [ ] Кнопка **Re-check** обновляет результат.
- [ ] Без OpenAI key поведение предсказуемо (ошибка, а не падение UI).

---

### Блок F. Поиск и чат внутри приложения

- [ ] **PII / граница OpenAI (FI-043):** в **логах / истории чата** в дашборде виден **оригинальный** текст вопроса (email, телефон и т.д., если пользователь их ввёл). На сторону OpenAI для эмбеддинга, основного ответа и валидации (FI-034) уходит **замаскированная** версия (`[EMAIL]`, `[PHONE]`, …). При политике «проверять утечки» — сверить по логам провайдера / прокси, что в prompt нет сырого PII.
- [ ] С **готовыми эмбеддингами** задать вопрос по содержимому загруженного документа — ответ **релевантен** (содержит смысл из документа, а не общие фразы).
- [ ] Вопрос **вне темы документа** — ответ корректно ограничен или честно говорит, что информации нет (зафиксируйте фактическое поведение как ожидаемое после согласования с продуктом).
- [ ] Пустой вопрос / слишком длинный — валидация или понятная ошибка.
- [ ] **Новая сессия** vs продолжение диалога (если в UI есть выбор) — проверить по ТЗ продукта.

---

### Блок G. Логи, обратная связь, разбор ответов

- [ ] Раздел **логов / сессий**: отображаются прошлые диалоги, открывается история сообщений.
- [ ] **Лайк / дизлайк** (или аналог) сохраняется; после перезагрузки состояние не сбрасывается неверно.
- [ ] Раздел **плохих ответов / review** (если есть): попадают ли туда сообщения с негативной оценкой; можно ли добавить «идеальный ответ» или заметку (как в текущем UI).

---

### Блок H. Отладка (Debug)

*Обычно для внутренних ролей; тестировщик проверяет, что не ломается UI.*

- [ ] Страница открывается только у подходящей роли (или у всех — уточнить у команды).
- [ ] Запрос выполняется, отображается режим (**vector / keyword / hybrid / none**; на проде Postgres обычно **hybrid**) и список фрагментов без ошибок JavaScript.

---

### Блок I. Админка (только admin)

- [ ] Пользователь **без** прав админа **не** видит пункт меню / получает 403 при прямом заходе.
- [ ] Пользователь **с** правами админа видит **сводные метрики** и список клиентов; цифры не «NaN» и страница не пустая при наличии данных.
- [ ] Нет утечки персональных данных других пользователей сверх задуманного в админке (сверить с политикой).

---

### Блок J. Виджет и публичный сценарий

- [ ] Страница **виджета** (iframe/демо) открывается, чат отправляет сообщение и получает ответ при корректном **client id / public id** (как настроено на стенде).
- [ ] **Встраивание**: фрагмент кода с дашборда вставлен на тестовую HTML-страницу — виджет грузится, CORS/безопасность не ломают сценарий на разрешённом домене.
- [ ] **Лимиты запросов** к публичному чату: при многократной отправке подряд (например 20+ сообщений в минуту) срабатывает ограничение — зафиксировать фактическое сообщение об ошибке (без DoS-атаки на прод).

---

### Блок K. Кросс-браузер и мобильная ширина

- [ ] Chrome / Safari / Firefox (хотя бы два браузера) — ключевые сценарии: вход, загрузка документа, один диалог в чате.
- [ ] Узкое окно (мобильная ширина): навигация доступна, кнопки нажимаются, нет горизонтального «слома» на главных экранах.

---

### Блок L. Негативные и граничные случаи (дополнительно)

- [ ] Обрыв сети посреди загрузки файла — сообщение об ошибке, повторная попытка возможна.
- [ ] Истёкшая сессия (долго не трогали вкладку) — при следующем действии просит войти снова.
- [ ] Два аккаунта в **двух окнах инкогнито**: документы и чаты **не пересекаются**.

---

## Часть 4. Рекомендуемый порядок первого прогона

1. **Блок A** (smoke)  
2. **B → C → D** (аккаунт, дашборд, документы)  
3. **E** (здоровье документа — по отдельному файлу)  
4. **F → G** (чат, логи, фидбек)  
5. **J** (виджет)  
6. **I, H** — если есть доступ  
7. **K, L** — по времени
