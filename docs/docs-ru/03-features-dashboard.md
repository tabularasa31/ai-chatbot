## Дашборд Chat9 — фичи и сценарии

Этот документ описывает основные разделы дашборда Chat9 с точки зрения администратора клиента. Здесь нет кода — только функциональность и пользовательские сценарии.

### Общая навигация

После входа в систему пользователь попадает в **приложение‑дашборд** с постоянным:
- верхним баром (бренд, email пользователя, выход);
- левым сайдбаром с разделами:
  - главные разделы (Dashboard, Knowledge, Logs/Review и др.);
  - блок **SETTINGS** (Agents, Response controls, Widget API и т. п.);
  - блок **Admin** (для пользователей с административными правами платформы).

Доступ к основным страницам дашборда защищён аутентификацией — без входа и валидного JWT‑токена они недоступны.

---

### 1. Регистрация, вход и верификация

**Цель:** чтобы клиент безопасно создал аккаунт и получил доступ к дашборду.

- **Регистрация**
  - Поля: email и пароль.
  - После отправки формы создаётся пользователь и связаный с ним клиент (tenant).

- **Подтверждение email**
  - На указанный адрес отправляется письмо с ссылкой.
  - До подтверждения:
    - пользователь может войти;
    - но для ряда операций с повышенными рисками (например, изменение ключей и конфигурации) требуется подтверждённый email.
  - В интерфейсе виден заметный баннер, напоминающий о необходимости верификации.

- **Вход**
  - Пользователь вводит email и пароль.
  - При успешном входе получает JWT‑токен, который автоматически используется фронтендом для API‑запросов дашборда.

- **Забыли пароль**
  - Пользователь вводит email, если забыл пароль.
  - Система отправляет письмо со ссылкой для сброса пароля (с ограничениями по числу запросов за период).
  - По ссылке пользователь задаёт новый пароль.

---

### 2. Главный экран / Dashboard

**Цель:** дать обзор состояния аккаунта и быстрый доступ к ключевым действиям.

Типично здесь присутствуют:
- краткий статус по OpenAI‑ключу (подключён/не подключён);
- `Your Client ID` (публичный `public_id`, формат `ch_…`) — отображается на странице `/dashboard` и используется в embed‑сниппете виджета (`embed.js?clientId=…`). Это не секретный API‑ключ.
- подсказка о следующем шаге (загрузить документы, встроить виджет и т. д.);
- быстрые ссылки на разделы:
  - Knowledge (база знаний),
  - Logs/Debug (диалоги и отладка),
  - Escalations (тикеты),
  - Settings (ключ, виджет, уровень ответов).

Для сейлз и саппорта важно: **Dashboard** — отправная точка, с которой удобно объяснять клиенту «где что лежит».

### 3. Knowledge — база знаний

**Цель:** собрать в одном месте все источники знаний клиента и управлять их индексированием для RAG‑поиска.

Раздел состоит из нескольких **подразделов/табов**:

#### 3.1. Документы (Documents)

Здесь клиент работает с файлами, загруженными в систему.

- **Поддерживаемые форматы:**
  - PDF‑документы (`.pdf`);
  - Markdown‑файлы (`.md`, `.mdx`);
  - Swagger/OpenAPI‑спецификации (`.json`, `.yaml`, `.yml`);
  - Word‑документы (`.docx`, `.doc`);
  - Текстовые файлы (`.txt`).

- **Как индексируются Swagger/OpenAPI**
  - raw JSON/YAML не эмбеддится напрямую;
  - система строит operation-aware чанки по `method + path`;
  - для больших endpoint'ов дополнительно создаются чанки со schema detail по request/response.

- **действия пользователя:**
  - загрузка документа;
  - просмотр списка и статусов (готов, в процессе индексации, ошибка и т. д.);
  - перезапуск индексации;
  - удаление документа.

- **Статусы индексации и health‑индикаторы**
  - Отображаются статусы процесса эмбеддинга (например: `ready`, `embedding`, `error`).
  - После индексации может отображаться оценка «здоровья» документа:
    - числовой балл или цветовой индикатор;
    - предупреждения (слишком мало текста, подозрительная структура и т. п.).

##### Типичный сценарий

1. Клиент загружает один или несколько файлов.
2. Запускается процесс индексации (может быть автоматическим или через кнопку).
3. В таблице видно прогресс и итоговый статус.
4. Если документ получает низкий health‑score, клиент может пересобрать его (например, подправить PDF или markdown).

#### 3.2. Источники по URL (URL sources)

Этот блок отвечает за индексацию **сайтов документации**.

- **Создание источника**
  - Клиент указывает базовый URL (например, документацию своего продукта).
  - Выбирает режим обновления (ежедневно, еженедельно, вручную).
  - Опционально задаёт исключающие паттерны путей, которые не нужно индексировать.

- **Поведение системы**
  - Сервис проверяет валидность URL и доступность ресурса.
  - Выполняется обход страниц в рамках заданного домена и глубины.
  - Каждая страница превращается в документ (с типом «URL»), который затем индексируется как обычный текстовый документ.
  - Источник имеет историю запусков и статусы (queued, indexing, ready, paused, error).

- **Управление ёмкостью**
  - Вся база знаний клиента (файлы + URL‑страницы) ограничена общим числом документов.
  - При достижении лимита новые страницы больше не добавляются, но уже существующие могут обновляться.

- **Управление страницами**
  - Клиент может удалять отдельные страницы из источника.
  - Удалённые страницы больше не будут индексироваться при последующих обновлениях — они добавляются в список исключений.

#### 3.3. Профиль и FAQ (Knowledge Profile / FAQ)

На более продвинутых этапах платформа поддерживает:

- **Профиль клиента (Tenant profile)**
  - Название продукта, извлечённые **topics** (темы из документации), support-контакты и glossary.
  - Используется для настройки контекста ответов, relevance/precheck и операторской настройки базы знаний.
  - Может автоматически извлекаться из документации и затем редактироваться клиентом.
  - Важно: `topics` здесь не означают строгие продуктовые модули. Это извлечённые темы документации, поэтому они могут соответствовать фичам, шагам настройки, страницам дашборда или другим повторяющимся темам.

- **FAQ**
  - Список часто задаваемых вопросов и подготовленных ответов.
  - Бот использует FAQ в двух режимах:
    - прямой ответ (если совпадение высокое и проходит защитные проверки);
    - добавление FAQ как подсказки в промпт (если полезно, но недостаточно уверенности для прямого ответа).
  - На дашборде есть интерфейс модерации FAQ:
    - просмотр кандидатов;
    - одобрение, редактирование, отклонение;
    - массовое одобрение.

### 4. Agents / OpenAI key — подключение модели

**Цель:** дать клиенту полный контроль над своим OpenAI‑ключом и расходами.

- Клиент указывает свой OpenAI API‑ключ в разделе **Agents** или аналогичном.
- Ключ:
  - хранится в зашифрованном виде;
  - используется только для запросов от конкретного клиента;
  - может быть заменён или отозван.
- При отсутствии ключа:
  - чат‑бот не сможет работать;
  - дашборд отображает заметный предупреждающий баннер;
  - основной пользовательский сценарий — сначала добавить ключ, потом переходить к настройке базы знаний и виджета.

Для сейлз важно: **платформа не продаёт свои токены OpenAI**, клиент всегда использует собственный ключ и оплачивает запросы напрямую в OpenAI.

### 5. Ответы и контроль раскрытия (Response controls)

**Цель:** дать клиенту возможность управлять тем, насколько «подробно» и «разговорно» отвечает бот.

В разделе **Response controls** клиент выбирает **уровень детализации** ответов, общий для всего аккаунта:

- примеры уровней:
  - **Detailed** — максимально подробные объяснения с примерами и разъяснениями.
  - **Standard** — баланс между полнотой и лаконичностью.
  - **Corporate** — более сдержанный, сухой стиль, минимум лишних деталей.

Этот выбор влияет:
- на системные подсказки для LLM;
- на формулировки ответов во всех каналах (дашборд, сервер‑к‑серверу, виджет).

### 6. Widget API и идентификация пользователей

**Цель:** дать клиенту удобный и безопасный способ встраивать виджет и, при необходимости, идентифицировать конечных пользователей.

Раздел **Widget / Widget API** содержит:

- **Сниппет для встраивания**
  - Готовая строка кода, которую клиент вставляет в сайт.
  - Включает публичный идентификатор клиента (безопасен для публикации).

- **Настройки идентификации (KYC‑подобный механизм)**
  - Клиент может включить режим, в котором виджет получает краткоживущий токен, подписанный секретом клиента.
  - Через этот токен можно безопасно передавать информацию о пользователе (например, ID в CRM), не раскрывая её напрямую в HTML.

- **Управление секретом**
  - В дашборде можно:
    - просмотреть статус секрета (есть/нет);
    - сгенерировать новый;
    - ротировать секрет (старый продолжает работать ограниченное время для плавного перехода).

Важно: в ответах бота в LLM‑промпт попадает строго ограниченный набор полей из этого контекста (тип тарифа, язык, сегмент аудитории и т. д.), а не все персональные данные.

### 7. Логи чатов, обратная связь и bad answers

**Цель:** дать клиенту прозрачный обзор того, как отвечает бот, и инструменты улучшения качества.

В разделе логов и отзывов обычно доступны:

- **История диалогов**
  - Список сессий, по каждой:
    - время начала и окончания;
    - количество сообщений;
    - оценка (если есть);
    - статус (был ли эскалирован).

- **Просмотр диалога**
  - По клику открывается детальная лента сообщений (пользователь ↔ бот).
  - Отображаются оценки «хороший/плохой ответ».

- **Bad answers / Review**
  - Фильтр по «плохим» ответам:
    - вопросы, на которые бот ответил неудовлетворительно;
    - ответы, помеченные пользователем или оператором.
  - Интерфейс, помогающий:
    - понять, какой контент не был найден или не был понят;
    - принять решение: добавить/исправить документ, создать FAQ, скорректировать формулировки.

Для сейлз полезно показывать этот раздел на демо: он демонстрирует **управляемость качества**, а не «магический чёрный ящик».

### 8. Эскалации (Escalations)

**Цель:** обеспечить плавный переход от бота к живому оператору при сложных запросах.

Раздел **Escalations** содержит:

- **Список тикетов**
  - Каждому тикету присваивается номер (например, `ESC-####` в рамках клиента).
  - В таблице видны:
    - статус (открыт/в работе/решён);
    - связанная чат‑сессия и последнее сообщение;
    - контактная информация пользователя (если доступна);
    - дата создания и обновления.

- **Детали тикета**
  - Полная цепочка сообщений до момента эскалации.
  - Причина эскалации (низкая релевантность, отсутствие документов, явный запрос «поговорить с человеком», ручное действие и т. п.).
  - Дополнительная информация для оператора (локаль, канал, идентификаторы).

- **Действия оператора**
  - взять тикет в работу;
  - пометить как решённый;
  - при необходимости — связаться с пользователем по внешнему каналу (email, чат поддержки клиента).

На стороне виджета пользователь видит, что его запрос передан человеку, и, в идеале, получает номер тикета или соответствующее уведомление.

### 9. Gap Analyzer

**Цель:** помочь клиенту понять, какие темы плохо покрыты документацией и какие вопросы от пользователей повторяются настолько часто, что их уже пора превращать в новые статьи, FAQ или onboarding-материалы.

Раздел **Gap Analyzer** в текущем продукте работает как отдельный операционный backlog и делится на два источника сигналов:

- **Mode A** — проблемы, найденные со стороны самой документации.
- **Mode B** — проблемы, найденные со стороны реальных пользовательских вопросов.

#### 9.1. Mode A — разрывы в документации

- Система смотрит на уже проиндексированную базу знаний клиента.
- Пытается найти темы, которые:
  - выглядят важными;
  - но покрыты слабо или фрагментарно.
- Если кандидат уже был dismissed, он не должен снова «всплывать» как новый активный gap просто из-за повторной индексации.

Для внутренних команд это полезно так:
- **сейлз** может показать клиенту, что платформа не только отвечает, но и находит слепые зоны в docs;
- **саппорт** может объяснить, почему один и тот же вопрос возникает снова и снова;
- **продакт / CS** получает список тем для улучшения help-center.

#### 9.2. Mode B — повторяющиеся реальные вопросы пользователей

- После завершения чатового запроса система может записать сигнал о проблемном вопросе.
- В ход идут ситуации вроде:
  - low confidence;
  - fallback;
  - reject;
  - escalation;
  - thumbs-down на ответ.
- Похожие вопросы собираются в кластеры.

Это значит, что клиент видит уже не «100 плохих сообщений по отдельности», а одну понятную проблему, например:
- «Как подключить SSO?»
- «Где настроить webhooks?»
- «Почему импорт CSV не работает?»

#### 9.3. Что видит клиент в интерфейсе

На странице `/gap-analyzer` клиент видит:

- верхние summary-card'ы по активным gap'ам;
- два основных блока: **Mode A** и **Mode B**;
- переключение между **Active view** и **Archive view**;
- фильтры по статусам;
- действия:
  - **Dismiss** — скрыть gap из активной работы;
  - **Reactivate** — вернуть в активную работу;
  - **Draft** — получить черновик текста/структуры для будущей статьи.

#### 9.4. Зачем нужен архив

Архив нужен не только для «мусора», но и для истории решений.

- В **Mode A** архив в основном означает dismissed docs-side темы.
- В **Mode B** архив хранит закрытые и dismissed кластеры пользовательских вопросов.

Это важно, потому что активный список должен оставаться коротким и приоритетным, а архив — сохранять контекст, что уже разбирали раньше.

#### 9.5. Как правильно объяснять Gap Analyzer клиенту

Не стоит обещать клиенту:
- что система автоматически перепишет ему документацию;
- что каждый найденный gap обязательно является багом продукта;
- что любой кластер Mode B — это точная «истина».

Правильная формулировка:

> Gap Analyzer — это рабочая очередь инсайтов. Он помогает понять, где пользователи застревают и чего не хватает в документации, но финальное решение всегда принимает человек.

Подробно внутреннюю логику см. в отдельном документе: [10-gap-analyzer.md](./10-gap-analyzer.md).

### 10. Debug / технический просмотр чат‑pipeline

**Цель:** дать технически подготовленным пользователям (и нашей команде) инструмент для глубокого анализа работы RAG‑pipeline.

В разделе **Debug**:

- Можно отправить тестовый запрос, который:
  - проходит через тот же pipeline, что и обычные чаты (инъекционный гард, поиск, FAQ, генерация, валидация);
  - но при этом не создаёт реальных записей в БД (используется чистый вычислительный путь).

- В ответе отображаются:
  - стратегия ответа (FAQ, RAG, отказ и т. п.);
  - причина возможного отказа;
  - сырая и финальная версия ответа;
  - подробные данные по retrieval‑слою (какие фрагменты были выбраны, какие метрики у них были);
  - итоговое решение по валидации.

Этот раздел редко нужен сейлз и саппорту, но крайне полезен разработчикам и продакт‑менеджерам для детального анализа качества.

### 11. Admin — метрики платформы (для наших админов)

**Цель:** мониторинг платформы в целом, а не отдельных клиентов.

Доступен только пользователям с флагом администратора платформы. Здесь могут отображаться:

- количество зарегистрированных пользователей и клиентов;
- суммарное число сессий и сообщений;
- объём использованных токенов (по платформе и по клиентам);
- технические метрики нагрузки.

Этот раздел не видят клиенты, но о его существовании полезно знать сотрудникам, которые работают с эксплуатацией и поддержкой.
