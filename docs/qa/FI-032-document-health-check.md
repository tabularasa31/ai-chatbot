# FI-032: проверка здоровья документа (Document Health Check)

## Описание фичи

После загрузки документа клиенты не всегда понимают, почему бот отвечает плохо. Часто дело в самом документе: слабая структура, нет контактов, обрывки разделов, нет примеров, устаревшие даты.

**Document Health Check** — это автоматический анализ текста документа (после успешного создания эмбеддингов) с помощью **GPT-4o-mini**. Модель возвращает структурированный результат: числовой **score** (0–100), время проверки **checked_at** и список **warnings** с типом, важностью и пояснением.

Проверка **не блокирует** загрузку и создание эмбеддингов: при сбое анализа документ остаётся обработанным, в БД может сохраниться запись с полем `error`.

**Где видно пользователю:** страница со списком документов — индикатор рядом с названием файла, подсказка при наведении, раскрывающийся список предупреждений, кнопка **Re-check** для повторного запуска проверки.

**API (для интеграций и отладки):**

- `GET /documents/{doc_id}/health` — текущий сохранённый результат (без повторного вызова GPT). Если проверки ещё не было — **404** с сообщением о недоступности.
- `POST /documents/{doc_id}/health/run` — синхронный перезапуск проверки; нужен настроенный OpenAI API key у клиента.

**Типы предупреждений (для справки):** отсутствие контактов, слабая структура, незавершённые разделы, нет примеров, устаревший контент. Важность: `low` / `medium` / `high` (на score влияют штрафы: −5 / −10 / −20, не ниже 0).

## Ручное тестирование (чеклист для тестировщика)

### Подготовка

- [ ] Есть аккаунт с **подтверждённой** электронной почтой.
- [ ] Создан клиент (организация/рабочее пространство).
- [ ] В настройках клиента указан **рабочий OpenAI API key** (без него эмбеддинги и кнопка повторной проверки не смогут вызвать модель).

### Основной сценарий

- [ ] Загрузить документ с осмысленным текстом (например `.md` или `.pdf`).
- [ ] Дождаться успешного завершения цепочки: документ в статусе готовности и созданы эмбеддинги (как в текущем UI после загрузки).
- [ ] У документа появился **индикатор здоровья** (не остаётся бесконечно в состоянии «Checking…»).
- [ ] Если есть предупреждения: при **наведении** на индикатор виден блок с текстом; под названием файла можно **раскрыть** список предупреждений.

### Соответствие подписи и цвета оценке

- [ ] **Good** (зелёная точка) при оценке **≥ 80**.
- [ ] **Fair** (жёлтая точка) при оценке **от 50 до 79 включительно**.
- [ ] **Needs attention** (красная точка) при оценке **ниже 50**.
- [ ] **Checking…** (серая точка), если результат проверки в интерфейсе ещё не получен (нет сохранённого `health_status`).
- [ ] **Unavailable**, если проверка отработала с ошибкой (нет числового score) — интерфейс не должен «ломаться».

### Кнопка «Re-check»

- [ ] Нажать **Re-check** у документа: кнопка корректно показывает процесс ожидания и затем обновляет индикатор/предупреждения.
- [ ] При отсутствии OpenAI key у клиента поведение предсказуемо (сообщение об ошибке, без падения страницы).

### API (по возможности: Postman, curl, DevTools)

- [ ] `GET /documents/{id}/health` **до** первой успешной записи результата — ответ **404**, текст про недоступность проверки.
- [ ] После появления результата в UI — тот же **GET** — **200**, в теле есть `checked_at`, массив `warnings`; при ошибке анализа может быть `error` и `score: null`.
- [ ] `POST /documents/{id}/health/run` с валидным JWT владельца и настроенным ключом — **200**, тело обновлённого статуса.

### Безопасность и изоляция

- [ ] Под **другим** пользователем (другой клиент) запрос к `GET` / `POST` health для **чужого** `document_id` — **404** (нет доступа к документу).

### Регрессия

- [ ] Загрузка, отображение списка и удаление документов работают как до фичи.
- [ ] Чат и поиск по базе знаний не ломаются из‑за проверки здоровья.
