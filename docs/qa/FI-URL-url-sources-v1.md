# FI-URL: URL Sources v1

## Что это за фича простыми словами

Раньше в **Knowledge** можно было загружать только файлы. Теперь можно добавить ещё и **сайт с документацией по ссылке**.

Пример: пользователь вставляет `https://docs.example.com`, а система:

1. проверяет, что ссылка выглядит нормальной;
2. пробует открыть сайт;
3. находит страницы на этом же сайте;
4. вытаскивает из них текст;
5. добавляет этот текст в базу знаний бота.

После этого бот должен уметь отвечать не только по загруженным файлам, но и по страницам сайта.

---

## Что тестировщик должен понимать заранее

- **URL source** = один источник по ссылке.
- **Knowledge** = страница в приложении, где видны все знания бота.
- **Indexing / crawl** = система обходит страницы сайта и сохраняет их.
- **Source detail** = экран с деталями одного URL source.

Если какие-то слова всё равно непонятны, это нормально. Для ручного теста важнее смотреть, что происходит в интерфейсе.

---

## Текущие лимиты фичи

Это важно знать заранее, чтобы не заводить баг там, где система работает как задумано.

- Разрешены только ссылки с `http://` и `https://`.
- Один source работает только **внутри одного домена**.
  Пример: если добавили `docs.example.com`, система не должна уходить на другой домен.
- Лимит: **до 50 страниц** на один URL source.
- Лимит глубины обхода: **до 3 уровней** от стартовой страницы.
- Доступные расписания: только **`daily`**, **`weekly`**, **`manual`**.
- Повторный **Refresh** можно запрашивать не чаще, чем **1 раз в час**.
- В detail-экране показываются только **последние 5 запусков**.
- Слишком большой ответ от сайта не должен индексироваться.
  Текущий лимит ответа: примерно **5 MB**.
- Если у клиента нет OpenAI key, source можно создать, но индексирование должно перейти в **paused**.

## Подготовка к тесту

- [ ] Есть аккаунт с **подтверждённым email**.
- [ ] Создан workspace / client.
- [ ] Для основного сценария у клиента указан **рабочий OpenAI API key**.
- [ ] Есть тестовый публичный сайт документации.
  Лучше маленький и понятный, чтобы было легко проверить результат.

---

## Основной сценарий: happy path

- [ ] Открыть страницу **Knowledge**.
- [ ] Найти блок / форму для добавления URL.
- [ ] Вставить валидную ссылку.
  Пример: `https://docs.example.com`
- [ ] Выбрать расписание.
- [ ] Нажать создание / сохранение source.
- [ ] Убедиться, что новый источник появился в списке.
- [ ] Проверить, что у источника есть понятный статус.
  Возможные статусы: `queued`, `indexing`, `ready`, `paused`, `error`.
- [ ] Дождаться завершения обработки.
- [ ] Убедиться, что успешный источник приходит в статус **`ready`**.

---

## Что должно быть видно в списке

- [ ] Видно имя source.
- [ ] Видно сам URL.
- [ ] Видно статус.
- [ ] Видно расписание.
- [ ] Видно, сколько страниц найдено и сколько проиндексировано.
- [ ] Если есть warning или error, они отображаются понятно и страница не ломается.

---

## Что проверить на detail-экране source

- [ ] Экран detail открывается без ошибки.
- [ ] Видны последние запуски source.
- [ ] Видны проиндексированные страницы.
- [ ] Видно число чанков / кусочков текста по страницам, если это предусмотрено UI.
- [ ] В списке запусков не больше **5 записей**.
- [ ] Самый новый запуск находится сверху.

---

## Refresh / edit / delete

### Refresh

- [ ] Кнопка **Refresh** запускает повторную обработку.
- [ ] После refresh статус меняется предсказуемо, а страница не зависает.
- [ ] Если нажать refresh слишком рано второй раз, система показывает понятное ограничение.
  Это нормально: сейчас refresh ограничен **1 разом в час**.

### Edit

- [ ] Можно изменить имя source.
- [ ] Можно изменить расписание.
- [ ] Можно изменить exclusions.
- [ ] После обновления страницы изменения не пропадают.

### Delete

- [ ] Source можно удалить.
- [ ] После удаления он исчезает из списка.
- [ ] Связанные URL-страницы тоже больше не должны отображаться как активные знания этого source.

---

## Негативные сценарии: простая валидация

- [ ] Ссылка без `http/https` не принимается.
- [ ] `ftp://...` не принимается.
- [ ] Явно сломанная ссылка показывает понятную ошибку.
- [ ] URL с логином или паролем внутри ссылки не принимается.
  Пример: `https://user:pass@example.com`
- [ ] Недопустимое расписание не принимается.
  Пример: `monthly`

---

## Негативные сценарии: опасные и локальные адреса

Это важный блок. Система не должна ходить во внутренние адреса сервера.

- [ ] `http://127.0.0.1/...` не принимается.
- [ ] `http://localhost/...` не принимается.
- [ ] Ссылка на приватный IP не принимается.
  Пример: `http://192.168.1.10/...`
- [ ] Если обычный публичный URL редиректит на `localhost` или приватный IP, source не должен индексироваться.
- [ ] Во всех этих случаях пользователь видит понятную ошибку, а UI не падает.

---

## Негативные сценарии: ошибки внешнего сайта

- [ ] Если сайт возвращает **404**, пользователь получает понятное сообщение, что страница не найдена.
- [ ] Если сайт возвращает **401** или **403**, пользователь получает понятное сообщение, что сайт недоступен для индексации.
- [ ] Если сайт возвращает **5xx**, пользователь получает понятное сообщение о временной проблеме сайта.
- [ ] Если ответ сайта слишком большой, система не ломается и показывает контролируемую ошибку.

---

## Сценарий без OpenAI key

- [ ] Удалить OpenAI key у клиента или использовать клиента без него.
- [ ] Создать URL source.
- [ ] Проверить, что source не ломает страницу.
- [ ] Проверить, что статус становится **`paused`**.
- [ ] Проверить, что есть понятный текст про отсутствие OpenAI key.
- [ ] После добавления ключа нажать **Refresh** и довести source до `ready`.

---

## Проверка прав доступа и изоляции

- [ ] Пользователь A создаёт URL source.
- [ ] Пользователь B под другим client не видит этот source в своём списке.
- [ ] Пользователь B не может открыть detail чужого source по прямой ссылке.
- [ ] Создание, редактирование, refresh и удаление доступны только подтверждённому пользователю.

---

## Регрессия: что не должно сломаться

- [ ] Обычная загрузка файлов в Knowledge работает как раньше.
- [ ] В одном списке корректно отображаются и файлы, и URL sources.
- [ ] Чат по-прежнему отвечает по уже загруженным файлам.
- [ ] После индексации URL source чат может отвечать и по страницам сайта.
- [ ] Если у пользователя нет client, страница ведёт себя предсказуемо и показывает понятную ошибку.

---

## Если что-то сломалось — что приложить к багу

Нужно приложить:

- URL, который тестировали;
- какой аккаунт использовали;
- был ли у клиента OpenAI key;
- что именно нажали по шагам;
- что ожидали увидеть;
- что увидели по факту;
- скрин списка или detail-экрана;
- если открывали DevTools → Network, приложить код ответа и текст ошибки.

---

## Очень короткий smoke-вариант

Если времени мало, проверь хотя бы это:

- [ ] можно создать URL source из валидной публичной ссылки;
- [ ] source появляется в Knowledge;
- [ ] source доходит до `ready`;
- [ ] detail-экран открывается;
- [ ] refresh работает;
- [ ] localhost / `127.0.0.1` не принимаются;
- [ ] без OpenAI key source уходит в `paused`, а UI не падает.
