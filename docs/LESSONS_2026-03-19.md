## Systematic Improvements for Future Sessions

### A. Prompt Creation Checklist

Before finalizing ANY Cursor prompt, verify:

```
□ Read actual code files (not just templates)
  - Check imports: `grep -n "^import\|^from" backend/models.py`
  - Verify function signatures: `grep -A5 "def function_name" backend/file.py`
  - Check current state: Is this already done? git log --oneline

□ Test code examples against reality
  - Would this import statement work? (dt.datetime vs datetime.datetime)
  - Are API responses real? (validate with actual route code)
  - Do test steps execute without errors? (curl commands, assertions)

□ One clear path per section
  - No "Option 1 / Option 2" unless there's explicit reason
  - One recommended approach, alternatives in NOTES only

□ CRITICAL WARNING section is present
  - SETUP with explicit MUST DO / DO NOT
  - Emphasize "git pull origin main"

□ Elle reviews BEFORE sending to Cursor
  - Give her all 4 prompts at once (if batch)
  - Ask: "Any issues before I send to Cursor?"
  - Incorporate feedback in single pass
```

### B. Code Verification Workflow

When writing ANY backend prompt:

```bash
# Step 1: Locate exact code
grep -rn "def function_name" backend/ --include="*.py"

# Step 2: Read context
cat backend/path/to/file.py | grep -A20 "def function_name"

# Step 3: Check imports
head -30 backend/path/to/file.py | grep "^import\|^from"

# Step 4: Verify it's not already done
git log --oneline --grep="keyword" | head -5

# Step 5: Draft prompt with real code
# (Copy exact function signature, real imports, etc.)

# Step 6: Test examples (in bash comments)
# Verify curl commands, assertions work
```

### C. Feedback Batch Model

Instead of:
```
Me: Create prompt X
Elle: [feedback on X]
Me: Update X
Elle: [more feedback on X]
```

Do:
```
Me: Create prompts X, Y, Z
Elle: [all feedback on X, Y, Z at once]
Me: Update all three based on feedback
Me: Send to Cursor
```

**Benefit:** Single pass, faster turnaround, more coherent changes.

### D. Code Review Protocol

When receiving CODE_REVIEW.md from client:

```
1. Parse ALL issues (don't just skim)
2. Categorize by priority (HIGH, MEDIUM, LOW)
3. Create spec/prompt for each priority level
4. Ask client: "Should I tackle HIGH first, or something else?"
5. Wait for answer before implementation

[Current mistake: assumed HIGH → MEDIUM → LOW order without asking]
```

### E. Architecture Decision Template

For new features like FI-CORS, use this before spec:

```
ARCHITECTURE DECISION

Problem:
- [What fails?]
- [Impact?]

Options:
1. [Quick hack]
   - Pros: X
   - Cons: Y

2. [Proper solution]
   - Pros: X
   - Cons: Y

Recommendation:
→ [Which? Why?]

Trade-offs:
- [What are we accepting/rejecting?]
```

**Why:** Makes decision transparent, helps avoid later regrets ("we should have done it properly").

## Updated Cursor Prompt Standards

### Mandatory Sections (in order):

```markdown
# [Feature Name]

⚠️ CRITICAL WARNING
[Must include git workflow / setup instructions]

[Exact commands with git pull origin main]

[Scope + Do NOT touch]

[Problem + Current code + Why]

## WHAT TO DO
[Step-by-step, one clear path]

## TESTING
[Checklist of what to verify]

## GIT PUSH
[Exact commands]

## NOTES
[Edge cases, security, caveats]
```

### Forbidden Patterns (remove from prompts):

- ❌ Multiple "Option 1 / Option 2" (pick one!)
- ❌ "You can use X or Y" (be prescriptive)
- ❌ Placeholder imports without verification
- ❌ Assumed API responses (check routes!)
- ❌ Vague instructions like "fix it appropriately" (be explicit)
