## Платформа Chat9 — архитектура без кода

Этот документ описывает архитектуру Chat9 на уровне компонентов и потоков данных, без деталей реализации и примеров кода. Его задача — дать общее понимание, как устроена система «под капотом».

### Крупные блоки архитектуры

- **Фронтенд**
  - Веб‑приложение дашборда (Next.js, React).
  - Маркетинговый лендинг.
  - Встраиваемый виджет (iframe + небольшой JS‑скрипт).

- **Бэкенд (API)**
  - HTTP‑API на базе FastAPI.
  - Обработка запросов дашборда, виджета и сервер‑к‑серверу.
  - Логика аутентификации, управления клиентами, документами, поиском, чат‑сессиями, эскалациями.

- **База данных**
  - PostgreSQL с расширением pgvector.
  - Хранение пользователей, клиентов, документов, эмбеддингов, чатов, эскалаций, настроек и т. д.

- **Внешние сервисы**
  - OpenAI API — генерация ответов и эмбеддингов.
  - Brevo — транзакционные email‑уведомления (регистрация, reset пароля и др.).
  - Система трассировки (Langfuse‑стиль) — для наблюдаемости за поиском и чат‑pipeline.

### Многотенантность и границы изоляции

Платформа мультиарендная: на одной инсталляции обслуживается множество независимых клиентов.

- **Tenant (Client)** — ключевая сущность для изоляции.
- Все основные данные (документы, эмбеддинги, чаты, эскалации, настройки) привязаны к конкретному клиенту.
- Любой запрос, связанный с данными, выполняется в контексте одного клиента:
  - По JWT (для дашборда и административных действий).
  - По X‑API‑Key (для сервер‑к‑серверу запросов).
  - По публичному идентификатору клиента (для публичного виджета).

Это гарантирует, что данные одного клиента не могут быть использованы для ответа другого.

### Взаимодействие основных компонентов

1. **Дашборд**
   - Работает в браузере администратора клиента.
   - Общается с бэкендом по HTTPS, используя JWT‑токен.
   - Вызывает API для:
     - загрузки и управления документами;
     - запуска индексации и просмотра статуса;
     - настройки FAQ, профиля, уровня раскрытия ответов и т. п.;
     - просмотра логов диалогов и эскалаций.

2. **Виджет**
   - Встраивается на сайт клиента в виде iframe + подключаемого скрипта.
   - Отправляет запросы на публичные API бэкенда без JWT:
     - инициализация сессии (с возможной идентификацией);
     - отправка сообщений;
     - эскалация к человеку.
   - Аутентификация виджета основана на публичном идентификаторе клиента и, опционально, краткоживущем токене идентификации пользователя (HMAC).

3. **Бэкенд и БД**
   - Бэкенд принимает HTTP‑запросы, проверяет авторизацию и клиентский контекст.
   - Для операций с данными использует ORM‑уровень (SQLAlchemy) поверх PostgreSQL.
   - Emбеддинги текстов хранятся в колонках pgvector, что позволяет выполнять быстрый семантический поиск.

4. **Внешние интеграции**
   - **OpenAI API**
     - Создание эмбеддингов для фрагментов документов и вопросов пользователей.
     - Генерация ответов и валидация (второй pass проверки «наземлённости» ответа на источники).
   - **Brevo**
     - Регистрация и подтверждение email.
     - Сброс пароля.
     - Уведомления по ряду продуктовых сценариев (например, эскалации).

### Поток данных: загрузка и индексация документа

1. Администратор клиента через дашборд загружает файл (PDF, Markdown, Swagger/OpenAPI) или добавляет источник по URL.
2. Бэкенд:
   - сохраняет исходный файл/источник и метадату;
   - извлекает текст из документа или страницы;
   - разбивает текст на перекрывающиеся фрагменты (чанки) с учётом типа документа;
   - ставит задачу на генерацию эмбеддингов в фоне (асинхронно).
3. Фоновая задача:
   - вызывает OpenAI API для каждого фрагмента;
   - сохраняет в БД векторное представление фрагмента и текстовый контекст;
   - обновляет статус документа (готов, ошибка и т. п.).
4. Дашборд отображает состояние в реальном времени (за счёт периодического опроса статуса документа).

Дополнительно после индексации может запускаться «проверка здоровья документа» — качественная оценка структуры, полноты и корректности текста с помощью LLM, с сохранением итогового балла и предупреждений.

### Поток данных: чат‑запрос конечного пользователя

1. Пользователь на сайте клиента пишет сообщение в виджет.
2. Виджет отправляет запрос на публичный чат‑endpoint бэкенда, идентифицируя:
   - клиента (по публичному ID);
   - сессию (уникальный идентификатор диалога);
   - опционально параметры локали и идентификации пользователя.
3. Бэкенд обрабатывает сообщение через единый **chat‑pipeline**:
   - редактирует/маскирует PII (персональные данные) до внешних вызовов;
   - выполняет гибридный поиск по базе знаний (векторный + BM25 + RRF);
   - принимает решения по FAQ и релевантности;
   - формирует промпт для LLM с контекстом фрагментов и историей диалога;
   - получает и при необходимости валидирует ответ отдельным LLM‑вызовом;
   - принимает решение об эскалации (создавать ли тикет для человека).
4. Ответ сохраняется в БД как часть сессии и возвращается во фронтенд виджета.

### Поиск и RAG на уровне архитектуры

Важная часть Chat9 — это **поисковый слой**, который обеспечивает RAG.

- **Векторный поиск (pgvector + HNSW)**
  - Каждый фрагмент документа хранится как вектор фиксированной размерности.
  - Для входящего вопроса строится вектор, и база ищет ближайшие фрагменты по косинусному расстоянию.

- **Гибридный поиск (BM25 + RRF)**
  - Помимо векторного сходства учитывается классический текстовый поиск по ключевым словам.
  - Две независимые ранжировки объединяются с помощью Reciprocal Rank Fusion, что повышает устойчивость к «редким» и строго формулируемым запросам (коды ошибок, артикулы, технические термины).

- **Пост‑обработка и надёжность**
  - Существуют механизмы оценки надёжности результата и работы с противоречиями в источниках.
  - Набор сигналов и политик позволяет, например, понижать доверие к ответам при обнаружении сильно противоречащих друг другу фрагментов.

### Наблюдаемость и трассировка

Для продакшн‑эксплуатации платформа ведёт детализированные **traces**:

- Для каждого поискового запроса и чат‑сообщения:
  - время работы разных стадий pipeline (эмбеддинги, поиск, генерация, валидация);
  - параметры гибридного поиска (режимы обработки запроса, количество вариантов, объём дополнительной работы);
  - дополнительные диагностические метрики.
- Есть флаги, управляющие объёмом собираемой информации (например, режим «полного захвата» трасс для отладки).

Эти данные используются:
- разработчиками — для анализа производительности и качества поиска;
- продуктовой командой — для оценки влияния изменений в retrieval‑политиках;
- саппортом — для диагностики инцидентов на конкретных клиентах.

### Инфраструктурный ландшафт

Типичная продакшн‑схема:

- Фронтенд (Next.js) развёрнут на отдельной платформе (например, Vercel).
- Бэкенд (FastAPI + PostgreSQL + pgvector) — на отдельном сервисе (например, Railway).
- Для развёртывания используется модель с двумя ветками:
  - `main` — ветка разработки.
  - `deploy` — ветка продакшна, с которой автоматически деплоится фронтенд и бекенд.
- GitHub Actions обеспечивает CI:
  - тесты и линтеры для бекенда;
  - линтер и сборку для фронтенда.

Для сотрудников важно понимать:
- фронтенд и бекенд развёрнуты раздельно и связаны по HTTPS;
- БД — единственный источник истины по данным клиентов;
- OpenAI и Brevo — внешние обязательные зависимости, от которых зависят ответы и почтовые сценарии.
