## KYC / идентификация пользователя в виджете

Этот документ описывает, как в Chat9 работает опциональная идентификация конечного пользователя в виджете. В коде эта функция называется **KYC** или **identified sessions**.

Актуально по состоянию на 29 апреля 2026 года.

### Зачем это нужно

По умолчанию виджет анонимный: Chat9 знает только, к какому тенанту относится чат, но не знает, кто именно из пользователей клиента пишет.

KYC позволяет тенанту:

- привязать чат к своему внешнему `user_id`;
- передать дополнительные признаки пользователя, например тариф или сегмент;
- улучшить маршрутизацию и приоритет эскалаций;
- персонализировать ответы без передачи лишних данных в промпт.

---

### Общая схема процесса

1. Клиент в дашборде генерирует секрет подписи для KYC.
2. Клиент хранит этот секрет на своём бэкенде.
3. Когда на сайте или в продукте открывается чат для конкретного пользователя, бэкенд клиента собирает payload и подписывает его.
4. Интеграция вызывает `POST /widget/session/init`.
5. Если токен валиден, Chat9 либо резюмирует подходящую identified-сессию этого пользователя, либо создаёт новую.
6. Chat9 сохраняет контекст пользователя в `chats.user_context` и ведёт lifecycle identified-пользователя в `contact_sessions` (`tenant_id + contact_id`).
7. Все следующие сообщения в чат идут уже с выданным `session_id`.

Важно: KYC отправляется именно на шаге `session/init`, а не на каждом сообщении.

Языковое следствие:

- до первого реального вопроса `locale` из KYC — это главный сигнал для greeting
  новой пустой сессии и других deterministic fallback-only ответов;
- если `locale` нет, следующей подсказкой становится `browser_locale`;
- если нет и его, безопасный дефолт — английский.

---

### Какой запрос отправляет тенант

В `POST /widget/session/init` передаются:

- `bot_id` — публичный идентификатор бота (формат `ch_...`, безопасно отдавать в HTML, не путать с приватным `api_key`);
- `identity_token` — опциональный подписанный токен с контекстом пользователя;
- `locale` — опциональная локаль браузера.

Пример тела запроса:

```json
{
  "bot_id": "ch_your_bot_public_id",
  "identity_token": "base64url_payload.hmac_signature",
  "locale": "ru-RU"
}
```

Тенант (владелец бота) и его KYC-секрет резолвятся бэкендом из `bot_id` — приватный `api_key` в браузер никогда не попадает.

---

### Что приходит в ответ

В ответ Chat9 возвращает только:

- `session_id`
- `mode`

Пример:

```json
{
  "session_id": "1c576fd8-cf10-4b58-a4e7-460ea0d19dbe",
  "mode": "identified"
}
```

Сами пользовательские данные обратно не возвращаются.

---

### Какие данные пользователя могут передаваться

Обязательные поля внутри `identity_token`:

- `user_id` — любая непустая строка после trimming;
- `exp` — время истечения токена (Unix timestamp в секундах).

Рекомендуемые:

- `iat` — время выпуска токена (Unix timestamp в секундах). Хранится в подписанном payload, но window-валидация не делается.

Опциональные поля:

- `email`
- `name`
- `plan_tier`
- `audience_tag`
- `company`
- `locale`

---

### Что означают важные поля

`user_id`

- Любая непустая строка.
- UUID не обязателен.
- Специального паттерна в коде нет.

`exp`

- Время, до которого токен считается действительным.
- Формат: Unix timestamp в секундах.

`iat`

- Время создания токена.
- Формат: Unix timestamp в секундах.

`audience_tag`

- Произвольный тег сегмента пользователя.
- Например: `vip`, `b2b`, `new_user`, `enterprise_admin`.
- Жёсткого справочника значений нет.

---

### Когда именно происходит отправка KYC

KYC происходит не:

- не при каждом сообщении;
- не внутри `POST /widget/chat`;
- не автоматически просто потому, что пользователь открыл страницу.

KYC происходит только тогда, когда интеграция сама вызывает `POST /widget/session/init` и передаёт `identity_token`.

Это важно для коммуникации с клиентами:

- стандартный публичный embed через the widget loader поддерживает KYC через `window.Chat9Config.identityToken`;
- если поле задано, the widget loader передаёт токен в widget через **постMessage-handshake** (виджет шлёт `chat9:ready` → the widget loader отвечает `chat9:identity`/`chat9:no-identity`); токен **не попадает в URL iframe**, чтобы не светиться в истории браузера, логах и `Referer`;
- если поле `null` или отсутствует, виджет уходит в anonymous-режим;
- альтернатива — кастомный bootstrap: клиент сам вызывает `POST /widget/session/init` с собранным token'ом из своего сервера.

Практически это означает:

- для stock embed без KYC greeting новой пустой сессии обычно будет ориентироваться на browser locale;
- для KYC-интеграции (через `Chat9Config.identityToken` или кастомный bootstrap) greeting и другие pre-question ответы должны сначала ориентироваться именно на `locale` из identity context.

---

### Что означают `mode = identified | anonymous`

`identified`

Это значит, что:

- токен был передан;
- подпись валидна (под секретом тенанта-владельца бота, резолвится по `bot_id` из запроса);
- токен не истёк;
- payload прошёл валидацию.

Последствия:

- Chat9 на этапе `session/init` либо резюмирует подходящий чат, либо создаёт новый;
- в `chats.user_context` сохраняется контекст пользователя;
- следующие сообщения используют этот контекст.

`anonymous`

Это значит, что:

- токен не был передан, или
- токен был передан, но не прошёл проверку.

Последствия:

- виджет продолжает работать;
- KYC-контекст не применяется;
- если передан только `locale`, Chat9 может сохранить `browser_locale`;
- если даже локаль не передана, чат обычно создаётся только на первом сообщении.

При этом язык deterministic-ответов до первого вопроса будет определяться так:
- `locale` из identity context;
- затем `browser_locale`;
- затем English.

---

### Что именно сохраняется у нас

Если токен прошёл проверку, в `chats.user_context` сохраняются прошедшие валидацию поля:

- `user_id`
- `email`
- `name`
- `plan_tier`
- `audience_tag`
- `company`
- `locale`

Дополнительно может быть добавлен:

- `browser_locale`

Служебные поля в контекст не сохраняются:

- `exp`
- `iat`

Для identified-пользователей дополнительно ведётся строка в `contact_sessions`:

- `tenant_id` (резолвится из `bot_id`)
- `contact_id` (это `user_id` из payload)
- лучшие известные identity-поля (`email`, `name`, `plan_tier`, `audience_tag`)
- `session_started_at`
- `session_ended_at`, если чат завершён
- `conversation_turns`

На пару `tenant_id + contact_id` допускается только одна активная запись `contact_sessions` (partial unique index `uq_contact_sessions_tenant_contact_active`, where `session_ended_at IS NULL`).

---

### Как теперь работает continuity / resume

Важно различать два механизма.

`anonymous`

- continuity работает только в рамках браузера;
- widget хранит `session_id` в `localStorage`;
- тот же браузер может продолжить чат в течение 24 часов;
- другой браузер или устройство получат новую сессию.

`identified`

- решение о resume принимает backend на `POST /widget/session/init`;
- ключ матчинга: `tenant_id + contact_id` (где `contact_id` — это `user_id` из payload);
- старая сессия резюмится только если:
  - `ended_at is null`;
  - последняя активность была не старше 24 часов.

Если условия не выполнены, создаётся новая чат-сессия и новая активная запись `contact_sessions`.

---

### Что значит «чат закрыт»

Закрытый чат — это чат, у которого в `chats.ended_at` стоит timestamp.

Практические последствия:

- закрытая сессия никогда не резюмится;
- если widget получает `chat_ended = true`, он очищает сохранённый `session_id`;
- пользователю показывается действие `Start new chat`;
- старая история остаётся на экране до reload;
- следующий вопрос запускает уже новую сессию ниже старого диалога.

---

### Как это влияет на ответы модели

В LLM-пайплайн попадает только безопасный поднабор полей:

- `plan_tier`
- `locale`
- `audience_tag`

Это значит:

- `user_id`, `email`, `name` не попадают напрямую в системный промпт;
- KYC используется для персонализации и внутренних процессов, но не как свободный PII-ввод в модель.

---

### Как это влияет на эскалации

При создании тикета эскалации система может использовать:

- `user_id`
- `email`
- `name`
- `plan_tier`

Поэтому identified-сессии полезны для саппорта:

- тикет можно привязать к пользователю клиента;
- можно лучше понимать приоритет;
- оператор видит больше полезного контекста.

---

### Что бывает при ошибках токена

Если токен невалиден, Chat9 не ломает виджет. Сессия просто становится `anonymous`.

Причины отказа могут быть такими:

- `malformed`
- `expired`
- `missing_user_id`
- `bad_signature`

Это полезно помнить саппорту: невалидный токен не вызывает падение виджета, а приводит к тихому откату в анонимный режим.

---

### Управление секретом

У клиента есть три dashboard/API действия:

- `POST /tenants/me/kyc/secret` — создать секрет;
- `GET /tenants/me/kyc/status` — посмотреть статус и masked hint;
- `POST /tenants/me/kyc/rotate` — ротировать секрет.

Особенности:

- полный секрет показывается только один раз;
- в базе он хранится зашифрованным;
- после ротации предыдущий секрет ещё 1 час остаётся валидным, чтобы клиент мог безболезненно обновить свои серверы.

---

### Коротко для внутренних команд

Для сейлз:

- KYC не обязателен для запуска виджета;
- это advanced-функция для клиентов, которым нужна привязка к своим пользователям и более умные эскалации.

Для саппорта:

- сначала уточнять, используют ли они просто the widget loader или кастомный bootstrap через `POST /widget/session/init`;
- если используют только стандартный embed, то KYC у них, скорее всего, вообще не включён в реальном потоке.

Для разработки:

- источник истины по поведению: `backend/widget/routes.py` (`widget_session_init`, `_resolve_widget_identity`), `backend/core/security.py` (HMAC signing/validation), `backend/auth/routes.py` (`/auth/me/widget-token` для cross-tenant случая), `backend/tenants/widget_chat_gate.py` (резолв тенанта по `bot_id`), `backend/chat/service.py`, `backend/escalation/service.py`.

---

### Cross-tenant KYC (dogfood support widget в дашборде Chat9)

KYC-секрет лежит **per-tenant** в `tenants.kyc_secret_key`. Для single-tenant интеграции (клиент ставит свой бот на свой сайт) тенант-юзер = тенант-владелец-бота, поэтому одного `generate_kyc_token(payload, secret)` на сервере клиента достаточно.

В дашборде Chat9 есть собственный embedded support widget, и это **cross-tenant** случай:

- Юзер из тенанта A (например ACME) залогинен в Chat9 dashboard;
- Бабл-чат внизу справа таргетит саппорт-бот, владелец которого — тенант B (Chat9, через env `NEXT_PUBLIC_CHAT9_BOT_ID`);
- Токен должен быть подписан секретом **B**, потому что `widget_session_init` валидирует подпись секретом тенанта-владельца-бота, не тенанта-юзера.

Это решено через `GET /auth/me/widget-token?bot_id=<bot_public_id>`:

- эндпоинт находит владельца бота через `get_bot_and_tenant_for_widget_session` (только `NOT_FOUND`/`INACTIVE`, без `NO_OPENAI` гейта — session/init не требует ключа OpenAI);
- подписывает payload секретом этого владельца;
- тот же секрет используется при валидации в `/widget/session/init`, поэтому identified-режим включается корректно.

Для backward compatibility при отсутствии `bot_id` в query токен подписывается секретом тенанта самого вызывающего юзера — это рабочий fallback для интеграций, где юзерский тенант сам владеет ботом.

Чтобы cross-tenant flow заработал на проде, у тенанта-владельца саппорт-бота должен быть установлен `kyc_secret_key`. Сгенерировать разово через **Dashboard → Settings → Widget API → Generate secret** под учёткой владельца. Без этого `/auth/me/widget-token?bot_id=...` вернёт 404 `"No identity secret configured."`.
