# Аналитика: дашборды и метрики (PostHog)

Chat9 использует PostHog для продуктовой аналитики. Все дашборды запинены в проекте **Chat9** и доступны команде.

---

## Структура дашбордов

| Дашборд | Назначение | Как часто смотреть |
|---|---|---|
| **Chat9 — Overview** | Основные SaaS-метрики: активность тенантов, объём разговоров | Ежедневно |
| **Chat9 — Answer Quality** | Качество ответов бота: отклонения, глубина разговора, FAQ | Еженедельно |
| **Chat9 — Reliability** | Здоровье OpenAI: ретраи, ошибки, latency | По алертам |
| **Chat9 — Language Pipeline** | Работа мультиязычности | По необходимости |
| **Chat9 — Tenant Activation Funnel** | Онбординг тенантов: от регистрации до первого чата | Еженедельно |

## Chat9 — Overview

Главный дашборд. Открываешь утром — за минуту видишь состояние продукта.

### Active tenants per week
Сколько уникальных тенантов имели хотя бы один чат-тёрн за неделю. Это главный сигнал роста и churn: если тенант пропал из графика на 2 недели — повод позвонить.

### Unique conversations per day
Уникальных сессий (`chat_id`) в день. Один разговор = одна сессия от начала до конца, независимо от количества сообщений. Это честная метрика вовлечённости: показывает сколько реальных людей пообщалось с ботами клиентов.

Соотношение разговоров и тёрнов (см. ниже) говорит о качестве взаимодействия. Много тёрнов на разговор = пользователи задают несколько вопросов (хороший знак). Мало тёрнов = уходят после первого сообщения.

### Chat turns — daily volume
Общее число обработанных сообщений в день по всем тенантам. Каждое входящее сообщение пользователя = 1 тёрн. Показывает нагрузку и общую активность платформы.

### Per-tenant table
Таблица для ежедневного мониторинга здоровья клиентов. Строки сортированы по убыванию разговоров.

| Колонка | Значение |
|---|---|
| `conversations` | Уникальных сессий за 30 дней — кто живой и насколько активный |
| `turns` | Общее число сообщений — объём нагрузки |
| `reject_pct` | % сообщений, которые бот отклонил (не ответил). Высокий % = база знаний дырявая или настройки топика плохие |
| `escalation_pct` | % сообщений, после которых бот передал чат человеку. Высокий % = бот не справляется самостоятельно |

**Норма:** `reject_pct` < 15%, `escalation_pct` < 20%. Всё выше — повод разобраться с конкретным тенантом.

### Silent tenants
Список тенантов, у которых **нет ни одного чат-тёрна за последние 14 дней**. Это самый ранний сигнал churn — тенант подключён, но бот молчит. Три типичные причины:
1. Виджет не установлен на сайт (техническая проблема)
2. Тенант отключил бота или сменил API-ключ
3. Продукт не прижился — повод для customer success звонка

Таблица выводит `tenant_id` и дату последнего тёрна. Если тенант новый (< 3 дней) — игнорируй, он ещё в онбординге.

## Chat9 — Answer Quality

Дашборд для понимания почему бот не справляется и насколько эффективно работает база знаний.

### Reject rate by reason
Breakdown отклонённых сообщений по причине. Три типа:

- **`injection`** — сообщение выглядит как попытка взломать промпт или обойти ограничения бота. Норма — единицы в день. Рост = кто-то целенаправленно атакует бота клиента.
- **`not_relevant`** — сообщение не по теме бота (настроенного тенантом домена). Частая причина: слишком узкий топик или пользователи задают общие вопросы. Решение — расширить настройки релевантности или базу знаний.
- **`low_retrieval`** — вопрос по теме, но бот не нашёл достаточно информации в базе знаний. Прямой сигнал дыры в документах. Решение — добавить материалы через Gap Analyzer.

### Injection attempt rate trend
Доля тёрнов с `reject_reason = injection` от общего числа тёрнов, по дням. Обычно близко к нулю. Аномальный рост — кто-то активно атакует бота конкретного тенанта. Смотри в связке с полем `tenant_id` чтобы определить цель.

### Escalation rate — trend
Процент тёрнов в день, завершившихся передачей чата живому оператору. Два вида эскалации:
- **Автоматическая** — бот понял что не справится (низкий скор схожести с базой знаний).
- **Ручная** — пользователь сам нажал «поговорить с человеком».

Рост тренда = сигнал для проверки базы знаний тенантов с высоким `escalation_pct` в Overview.

### FAQ hit rate
Доля тёрнов, обработанных через FAQ-стратегию (`faq_direct` или `faq_context`), по дням. FAQ — самый быстрый и точный путь ответа. Высокий FAQ hit rate = база знаний хорошо структурирована, бот находит готовые ответы без LLM-генерации.

Падение FAQ hit rate при неизменном объёме тёрнов = пользователи стали задавать новые типы вопросов, которых нет в FAQ. Повод заглянуть в Gap Analyzer.

### Avg conversation depth per tenant
Среднее число тёрнов на один `chat_id` по каждому тенанту. Показывает насколько глубоко пользователи вовлечены.

- **Глубина 1–2** — пользователи уходят после первого-второго сообщения. Либо бот сразу даёт нужный ответ (хорошо), либо сразу разочаровывает (плохо). Смотри в связке с `reject_pct`.
- **Глубина 3–5** — норма. Пользователь уточняет, бот отвечает.
- **Глубина > 7** — пользователь «застрял» — не может получить нужный ответ и продолжает переспрашивать. Повод для анализа конкретных сессий.

### Reliability score distribution
Таблица breakdown по значениям `reliability_score` (`low`, `medium`, `high`) в разрезе тенантов.

| Уровень | Что означает |
|---|---|
| `high` | Бот нашёл хорошие источники, ответ уверенный |
| `medium` | Источники найдены, но уверенность умеренная |
| `low` | Бот отвечал с недостаточной поддержкой из базы — ответ мог быть неточным |

Тенанты с высокой долей `low` — приоритет для обогащения базы знаний.

## Chat9 — Reliability

Инженерный дашборд. Смотрите при инцидентах или алертах.

### Response latency p50/p95
Медиана (p50) и 95-й перцентиль (p95) `latency_ms` по дням — время от получения сообщения до отправки ответа.

- **p50** — «типичный» пользовательский опыт. Норма: < 3 000 мс.
- **p95** — «худший» опыт 5% пользователей. Норма: < 8 000 мс.

Рост p95 при стабильном p50 = отдельные медленные запросы (большие документы, длинные промпты). Рост обоих = системная проблема (нагрузка на OpenAI, медленный retrieval).

`latency_ms` — end-to-end время (гарды + retrieval + LLM + запись в БД). Для детального анализа: `retrieval_ms` и `llm_ms` доступны в тех же событиях `chat.turn` и позволяют понять где именно теряется время.

### OpenAI Retries — attempts vs exhausted
- **Attempts** — каждый повторный запрос к OpenAI после ошибки (rate limit, timeout, 5xx). Один инцидент может дать несколько attempts.
- **Exhausted** — все попытки исчерпаны, пользователь получил ошибку.

В норме `exhausted` в разы меньше `attempts`. Если линии сближаются — OpenAI нестабилен или количество ретраев недостаточно.

### OpenAI exhausted rate
Одна цифра: процент цепочек ретраев, которые завершились ошибкой для пользователя (`exhausted / attempts × 100%`). Норма: < 5%. При > 10% — инцидент.

## Chat9 — Language Pipeline

Дашборд для диагностики мультиязычности.

### Language — resolved per day
Сколько раз в день успешно определился язык сообщения. Косвенный прокси для числа обработанных сообщений.

### Language — switched & fallback
- **Switched** — пользователь написал на другом языке, и бот переключился. Нормальное поведение, мультиязычность работает.
- **Detect fallback** — детекция языка не удалась (слишком короткое сообщение, смешанный текст) и использовался язык по умолчанию. Рост = много очень коротких сообщений или проблема в библиотеке детекции.

### Language distribution
Breakdown сообщений по языку браузера. Показывает географию аудитории тенантов.

## Chat9 — Tenant Activation Funnel

Онбординг-дашборд. Показывает конверсию тенантов по воронке активации.

### Activation funnel
Четыре шага воронки в порядке онбординга:

| Шаг | Событие | Что означает |
|---|---|---|
| 1 | `tenant.created` | Тенант зарегистрировался и создал аккаунт |
| 2 | `knowledge.uploaded` | Загрузил хотя бы один документ или URL-источник |
| 3 | `bot.created` | Создал бота |
| 4 | `chat.turn` | Первый реальный чат-тёрн через бота |

Смотри drop-off между шагами:
- Большой drop между **1 → 2**: тенанты не понимают что надо загрузить базу знаний. Повод улучшить онбординг-подсказки.
- Большой drop между **2 → 3**: создание бота вызывает затруднения. Повод упростить форму.
- Большой drop между **3 → 4**: бот создан, но виджет не установлен на сайт или нет трафика. Повод написать тенанту с инструкцией.

Тенанты, застрявшие на шаге 1–2 > 7 дней — горячие лиды для customer success: они зарегистрировались, но не завершили активацию.

## Как данные попадают в PostHog

### Событие `chat.turn`

Каждый обработанный запрос пользователя генерирует событие `chat.turn`:

| Поле | Описание |
|---|---|
| `tenant_id` | Идентификатор тенанта |
| `bot_id` | Идентификатор бота |
| `chat_id` | ID сессии (для подсчёта уникальных разговоров) |
| `strategy` | Путь обработки: `rag_only`, `faq_direct`, `faq_context`, `guard_reject`, `capability_response` |
| `is_reject` | Было ли сообщение отклонено |
| `reject_reason` | Причина отклонения: `injection`, `not_relevant`, `low_retrieval` |
| `escalated` | Была ли инициирована эскалация |
| `latency_ms` | Общее время обработки в миллисекундах (end-to-end: от входящего сообщения до ответа) |
| `retrieval_ms` | Время векторного поиска в базе знаний (0 если retrieval не выполнялся: guard_reject до retrieval, faq_direct) |
| `llm_ms` | Время генерации ответа OpenAI (0 если LLM не вызывался: guard_reject, faq_direct) |
| `reliability_score` | Качество RAG-ретривала: `low` / `medium` / `high` |
| `best_confidence_score` | Числовой скор схожести с базой знаний (0–1) |

### Activation events

| Событие | Когда | Поля |
|---|---|---|
| `tenant.created` | Создание нового тенанта | `tenant_id` |
| `knowledge.uploaded` | Загрузка файла или URL-источника | `tenant_id`, `source_type` (`file`/`url`), `file_type` (для файлов) |
| `bot.created` | Создание нового бота | `tenant_id`, `bot_id` |

> **Важно:** все события не генерируются в тестовом окружении (`environment=test`) и не отправляются если не настроен `POSTHOG_API_KEY`.
