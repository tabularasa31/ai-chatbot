# FI-URL: URL Sources v1

**Фича:** пользователь открывает **Knowledge**, добавляет ссылку на сайт с документацией, система проверяет ссылку, обходит страницы этого сайта, сохраняет текст в базу знаний и показывает результат в интерфейсе. После успешной обработки бот должен уметь отвечать по этим страницам так же, как по обычным загруженным файлам.

## Что это за фича простыми словами

Раньше в **Knowledge** можно было загружать только файлы. Теперь можно добавить ещё и **сайт с документацией по ссылке**.

Пример: пользователь вставляет `https://docs.example.com`, а система:

1. проверяет, что ссылка выглядит нормальной;
2. пробует открыть сайт;
3. находит страницы на этом же сайте;
4. вытаскивает из них текст;
5. добавляет этот текст в базу знаний бота.

После этого бот должен уметь отвечать не только по загруженным файлам, но и по страницам сайта.

---

## Как эта фича должна работать в нормальном процессе

Ниже описан обычный, “здоровый” сценарий без ошибок.

1. Пользователь открывает страницу **Knowledge**.
2. Пользователь вставляет URL сайта документации.
3. Пользователь выбирает расписание и при необходимости заполняет exclusions.
4. Пользователь сохраняет source.
5. Новый source появляется в списке на странице Knowledge.
6. Система начинает обработку:
   сначала проверяет ссылку, потом ищет страницы на этом сайте, потом читает текст и сохраняет его.
7. Во время обработки source может быть в статусе **`queued`** или **`indexing`**.
8. Если всё прошло успешно, source приходит в статус **`ready`**.
9. После этого в строке source видно основную информацию:
   статус, расписание, прогресс индексации, здоровье / warning и основные действия.
10. При нажатии на source под его строкой раскрывается detail-блок:
   последние запуски, exclusions и связанные предупреждения / ошибки.
11. Если потом сайт изменился, пользователь может нажать **Refresh**, чтобы заново обновить этот источник.

Что важно понимать:

- пользователь работает не с “отдельными страницами”, а с **одним source**;
- внутри этого source система сама находит и обрабатывает страницы сайта;
- если всё работает нормально, итоговый статус должен быть **`ready`**;
- если что-то пошло не так, пользователь должен увидеть понятную ошибку или понятный статус, а не сломанную страницу.
- поле **exclusions** не обязательно заполнять;
- если сайт маленький и доступный, чаще всего путь статусов такой:
  **`queued` -> `indexing` -> `ready`**.

## Что тестировщик должен понимать заранее

- **URL source** = один источник по ссылке.
- **Knowledge** = страница в приложении, где видны все знания бота.
- **Indexing / crawl** = система обходит страницы сайта и сохраняет их.
- **Source detail** = раскрывающийся блок под строкой одного URL source в списке Knowledge.

### Простое объяснение терминов

- **Source** = одна запись в Knowledge, которая привязана к одному сайту или одной ссылке.
  Проще говоря: добавили один URL -> получили один source.
- **Refresh** = кнопка, которая просит систему заново проверить сайт и обновить данные.
  Это нужно, если страницы на сайте изменились или если первая обработка прошла с ошибкой.
- **Schedule** = как часто система должна автоматически возвращаться к этому source и проверять его снова.
  Сейчас доступны такие варианты:
  - `manual` = только вручную по кнопке **Refresh**
  - `daily` = примерно раз в день
  - `weekly` = примерно раз в неделю
- **Exclusions** = необязательное поле со списком путей, которые нужно пропустить и не индексировать.
  Это нужно, если на сайте есть лишние разделы, которые не должны попадать в базу знаний.
  Примеры:
  - `/blog/*` = не брать страницы блога
  - `/changelog/*` = не брать changelog
  - `/private/*` = не брать закрытый раздел документации

### Что значат статусы source

- **`queued`** = источник создан и поставлен в очередь на обработку.
  Это нормальный короткий промежуточный статус сразу после создания или после refresh.
- **`indexing`** = система прямо сейчас обходит страницы сайта и сохраняет их.
  Это тоже нормальный рабочий статус.
- **`ready`** = обработка завершилась успешно.
  Это основной успешный финальный статус.
- **`paused`** = система не может продолжить работу, пока пользователь что-то не исправит.
  Самый частый пример: у клиента нет OpenAI key.
- **`error`** = обработка завершилась с проблемой.
  Например: сайт недоступен, слишком много страниц не удалось открыть, или произошла другая ошибка.

Что важно для теста:

- статус **`queued`** или **`indexing`** не должен висеть бесконечно без причины;
- статус **`ready`** означает, что source обработан успешно;
- статус **`paused`** обычно требует действия от пользователя;
- статус **`error`** должен сопровождаться понятным сообщением, а не просто “сломаться молча”.

Что обычно **не** считается багом:

- source короткое время стоит в `queued` сразу после создания;
- source какое-то время стоит в `indexing`, пока сайт обходится;
- после нажатия **Refresh** source снова временно возвращается в `queued` или `indexing`;
- если у пользователя нет OpenAI key, source не дойдёт до `ready`, а перейдёт в `paused`.

Если какие-то слова всё равно непонятны, это нормально. Для ручного теста важнее смотреть, что происходит в интерфейсе.

## Текущие лимиты фичи

Это важно знать заранее, чтобы не заводить баг там, где система работает как задумано.

- Разрешены только ссылки с `http://` и `https://`.
- Один source работает только **внутри одного домена**.
  Пример: если добавили `docs.example.com`, система не должна уходить на другой домен.
- Общий лимит knowledge: **до 100 документов на клиента**.
  Сюда входят и загруженные файлы, и URL-страницы из всех sources вместе.
- Лимит глубины обхода: **до 3 уровней** от стартовой страницы.
- Доступные расписания: только **`daily`**, **`weekly`**, **`manual`**.
- Повторный **Refresh** можно запрашивать не чаще, чем **1 раз в час**.
- В detail-блоке показываются только **последние 5 запусков**.
- Слишком большой ответ от сайта не должен индексироваться.
  Текущий лимит ответа: примерно **5 MB**.
- Если у клиента нет OpenAI key, source можно создать, но индексирование должно перейти в **paused**.

## Основной сценарий: happy path

- [ ] Открыть страницу **Knowledge**.
- [ ] Найти блок / форму для добавления URL.
- [ ] Вставить валидную ссылку.
  Пример: `https://docs.example.com`
- [ ] Выбрать расписание.
  Если не хочется проверять автоповторные обновления, удобнее брать `manual`.
- [ ] Нажать создание / сохранение source.
- [ ] Убедиться, что новый источник появился в списке.
- [ ] Проверить, что у источника есть понятный статус.
  Возможные статусы: `queued`, `indexing`, `ready`, `paused`, `error`.
- [ ] Дождаться завершения обработки.
- [ ] Убедиться, что успешный источник приходит в статус **`ready`**.
- [ ] После успешной обработки убедиться, что в строке source корректно отображаются indexed count, schedule и health.
- [ ] Открыть detail-блок под строкой source и убедиться, что внутри есть последние запуски и exclusions.

## Что должно быть видно в списке

- [ ] Видно имя source.
- [ ] Видно сам URL.
- [ ] Видно статус.
- [ ] Видно расписание.
- [ ] Видно, сколько страниц найдено и сколько проиндексировано.
- [ ] Виден health / warning статус source.
- [ ] Видны действия **Edit**, **Refresh**, **Delete**.
- [ ] Если есть warning или error, они отображаются понятно и страница не ломается.
- [ ] Если source ещё обрабатывается, в списке всё равно не должно быть “битой” строки или пустых полей без объяснения.

## Что проверить в detail-блоке source

- [ ] Detail-блок открывается прямо под строкой source без ошибки.
- [ ] Повторный клик по source сворачивает detail-блок.
- [ ] Видны последние запуски source.
- [ ] Видны проиндексированные страницы source.
- [ ] Видны exclusions source.
- [ ] В списке запусков не больше **5 записей**.
- [ ] Самый новый запуск находится сверху.

## Refresh / edit / delete

### Refresh

- [ ] Кнопка **Refresh** запускает повторную обработку.
- [ ] После refresh статус меняется предсказуемо, а страница не зависает.
- [ ] Если нажать refresh слишком рано второй раз, система показывает понятное ограничение.
  Это нормально: сейчас refresh ограничен **1 разом в час**.

### Edit

- [ ] Можно изменить имя source.
- [ ] Можно изменить расписание.
- [ ] Можно изменить exclusions.
  Для проверки удобно ввести что-то простое, например `/blog/*`.
- [ ] Можно сохранить не больше `50` exclusion patterns.
- [ ] Один exclusion pattern длиннее `255` символов не принимается с понятной ошибкой.
- [ ] После обновления страницы изменения не пропадают.
- [ ] Если указан exclusion, страницы из этого пути не должны попадать в результат при следующем crawl / refresh.

### Delete

- [ ] Source можно удалить.
- [ ] После удаления он исчезает из списка.
- [ ] Связанные URL-страницы тоже больше не должны отображаться как активные знания этого source.

## Негативные сценарии: простая валидация

- [ ] Ссылка без `http/https` не принимается.
- [ ] `ftp://...` не принимается.
- [ ] Явно сломанная ссылка показывает понятную ошибку.
- [ ] URL с логином или паролем внутри ссылки не принимается.
  Пример: `https://user:pass@example.com`
- [ ] Недопустимое расписание не принимается.
  Пример: `monthly`
- [ ] Если exclusions заполнены странным текстом, страница не должна падать.
  Даже если правило не сработало как ожидалось, UI не должен ломаться.
- [ ] Если exclusions больше 50, запрос не должен проходить.
- [ ] Если один exclusion слишком длинный, запрос не должен проходить.

---

## Source detail / run history contract

- [ ] В `recent_runs[].failed_urls` каждый элемент содержит поля `url` и `reason`.
- [ ] В detail API не появляется произвольная форма failed item без этих двух полей.

---

## Негативные сценарии: опасные и локальные адреса

Это важный блок. Система не должна ходить во внутренние адреса сервера.

- [ ] `http://127.0.0.1/...` не принимается.
- [ ] `http://localhost/...` не принимается.
- [ ] Ссылка на приватный IP не принимается.
  Пример: `http://192.168.1.10/...`
- [ ] Если внешне ссылка выглядит обычной и публичной, но потом автоматически перебрасывает на `localhost` или приватный IP, source всё равно не должен индексироваться.
  Проще говоря: нельзя обойти защиту через “скрытый” редирект.
- [ ] Во всех этих случаях пользователь видит понятную ошибку, а UI не падает.

---

## Негативные сценарии: ошибки внешнего сайта

- [ ] Если сайт возвращает **404**, пользователь получает понятное сообщение, что страница не найдена.
- [ ] Если сайт возвращает **401** или **403**, пользователь получает понятное сообщение, что сайт недоступен для индексации.
- [ ] Если сайт возвращает **5xx**, пользователь получает понятное сообщение о временной проблеме сайта.
- [ ] Если ответ сайта слишком большой, система не ломается и показывает контролируемую ошибку.

---

## Сценарий без OpenAI key

- [ ] Удалить OpenAI key у клиента или использовать клиента без него.
- [ ] Создать URL source.
- [ ] Проверить, что source не ломает страницу.
- [ ] Проверить, что статус становится **`paused`**.
- [ ] Проверить, что есть понятный текст про отсутствие OpenAI key.
- [ ] После добавления ключа нажать **Refresh** и довести source до `ready`.
  Это важная проверка: source не должен навсегда остаться в `paused`, если проблема уже исправлена.

---

## Проверка прав доступа и изоляции

- [ ] Пользователь A создаёт URL source.
- [ ] Пользователь B под другим client не видит этот source в своём списке.
- [ ] Пользователь B не может открыть detail чужого source по прямой ссылке.
- [ ] Создание, редактирование, refresh и удаление доступны только подтверждённому пользователю.

---

## Регрессия: что не должно сломаться

- [ ] Обычная загрузка файлов в Knowledge работает как раньше.
- [ ] В одном списке корректно отображаются и файлы, и URL sources.
- [ ] Общий лимит knowledge считается суммарно для файлов и URL pages, а не отдельно по каждому source.
- [ ] Чат по-прежнему отвечает по уже загруженным файлам.
- [ ] После индексации URL source чат может отвечать и по страницам сайта.
- [ ] Если у пользователя нет client, страница ведёт себя предсказуемо и показывает понятную ошибку.

---

## Если что-то сломалось — что приложить к багу

Нужно приложить:

- URL, который тестировали;
- какое расписание выбрали;
- были ли указаны exclusions;
- какой аккаунт использовали;
- был ли у клиента OpenAI key;
- что именно нажали по шагам;
- что ожидали увидеть;
- что увидели по факту;
- скрин списка или detail-блока;
- если открывали DevTools → Network, приложить код ответа и текст ошибки.

## Очень короткий smoke-вариант

Если времени мало, проверь хотя бы это:

- [ ] можно создать URL source из валидной публичной ссылки;
- [ ] source появляется в Knowledge;
- [ ] source доходит до `ready`;
- [ ] detail-блок под source открывается;
- [ ] refresh работает;
- [ ] localhost / `127.0.0.1` не принимаются;
- [ ] без OpenAI key source уходит в `paused`, а UI не падает.
