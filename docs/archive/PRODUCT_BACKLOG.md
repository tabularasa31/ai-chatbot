## RICE-приоритизация (2026-03-18)

RICE = (Reach × Impact × Confidence) / Effort

| Метрика | Шкала |
|---------|-------|
| Reach | 1–10 (кол-во затронутых пользователей/клиентов) |
| Impact | 1=мало, 3=средне, 5=высокая |
| Confidence | % уверенности в оценках |
| Effort | дни разработки |

#### [FI-005] Приветственное сообщение в виджете
**RICE: 1440** — лидер

**Что:**
- В настройках клиента хранится строка `greeting_message`.
- При открытии виджета бот показывает это сообщение первым (роль assistant).
- Если не задано — стандартный шаблон ("Привет, я бот поддержки...").

**Ценность:**
- Каждый конечный пользователь виджета видит это сразу.
- Устраняет "страх пустого чата".
- Пользователь сразу понимает, с чем может помочь бот.

**Effort:** 1 день
**Files:** `Client` model, dashboard Settings, `embed.js`

### 🟠 P2 (западный рынок)

#### [FI-027] Ticketing systems integration (Zendesk, Intercom, Freshdesk)

**RICE:** ~600 (высокий Impact для западной аудитории, средний Reach сейчас)

**Идея:**
Вместо ручного экспорта тикетов — прямая интеграция с тикетными системами.

**Три уровня реализации:**

1. **Импорт тикетов как база знаний (читать)**
   - Клиент подключает Zendesk/Intercom через OAuth или API key.
   - Мы автоматически тянем закрытые тикеты с ответами саппорта.
   - Превращаем в эмбеддинги → бот сразу знает историю поддержки.
   - Синхронизация периодически (раз в день/неделю).
   - **УТП:** "Подключи Zendesk — бот обучается на всей истории."

2. **Escalation (писать)**
   - Если бот не знает → автоматически создаёт тикет в Zendesk.
   - Пользователь видит: "Ваш вопрос передан в поддержку."
   - Замыкает петлю: бот → человек → новый тикет в базе знаний.

3. **Live handoff**
   - Плавный переход к живому агенту через Intercom/Zendesk chat.

**Приоритет интеграций:**
- Zendesk (самый популярный в B2B SaaS)
- Intercom
- Freshdesk
- HubSpot

**Ценность для позиционирования:**
- Сейчас: "Загрузи доки → получи бота."
- С интеграцией: "Подключи Zendesk → бот знает всё и передаёт нерешённые вопросы обратно."

**Effort:** 5–8 дней (Level 1), 10–15 дней (Level 1+2)

### 🔴 P1 (данные и промпт — до алгоритмов)

#### [FI-031] Org config layer

**Идея:**
Статичные данные клиента (email поддержки, контакт менеджера, тестовый период, домены) хранятся отдельно от RAG и всегда вставляются в system prompt.

**Проблема которую решает:**
Бот не знает email поддержки, тестового периода, контактов — потому что их нет в документах. Это не проблема RAG, это отсутствие статичных данных.

**Что сделать:**
- Поле `org_config` (JSON) на модели Client.
- UI в дашборде: форма для заполнения (support_email, account_manager, trial_period, support_url и др.).
- В `build_rag_prompt()` — вставлять org_config блоком перед контекстом.

**Effort:** 2 дня
**Делать до FI-007** (system prompt без org config будет неполным).

---

#### [FI-032] Document health check (анализ качества документов)

**Идея:**
После загрузки документа — автоматический GPT-анализ и отчёт клиенту о проблемах.

**Что анализируем:**
1. **Неполные разделы** — есть вопрос но нет чёткого ответа.
   - "В разделе 'Защита потока' не указан email для подключения."
2. **Устаревшие данные** — даты, версии, URL.
   - "Найдена ссылка example.trbcdn.net — она не отвечает. Обновите или удалите."
3. **Пробелы в документации** — на основе типичных вопросов для этой категории продукта.
   - "Нет раздела о тестовом периоде — это частый вопрос для CDN-сервисов."
4. **Качество для RAG** — слишком длинные разделы без структуры, нет заголовков.
   - "Раздел 'Настройка' содержит 3000 слов без подзаголовков — рекомендуем разбить."

**Реализация MVP:**
- После загрузки → фоновый GPT-анализ (BackgroundTasks).
- В дашборде: таб/секция "Document health" с предупреждениями по категориям.
- Клиент видит: что улучшить → улучшает → бот отвечает точнее.

**Ценность для продукта:**
- УТП: "Не просто загрузи доки — мы скажем где они неполные."
- Меньше жалоб на качество ответов.
- Клиент сам улучшает данные без нашего участия.

**Effort:** 3 дня

### 🔴 P1 (learning loop)

#### [FI-037] Approved Answers Layer (👍 → instant answers)

**Идея:**
Ответы, помеченные 👍, становятся отдельным слоем поиска. Перед RAG — ищем в проверенных ответах. Если нашли → отдаём без GPT.

**Три уровня реализации:**

**Уровень 1 — MVP:**
- Таблица `approved_answers` (вопрос + ответ + tenant_id + embedding).
- При каждом 👍 → запись в `approved_answers`.
- Перед RAG: semantic search по `approved_answers`.
- Если similarity > 0.85 → отдаём готовый ответ (без GPT).

**Уровень 2 — Редактирование:**
- Использовать `ideal_answer` из `Message` как эталон вместо исходного ответа.
- Клиент редактирует ответ в `/review` → "Одобрить как эталонный".
- Версионирование approved answers.

**Уровень 3 — Fine-tuning:**
- (вопрос + контекст + ideal_answer) → датасет для fine-tuning.
- Бот обучается на своих лучших ответах без переобучения модели вручную.

**Что это даёт:**
- **Точность:** проверенные ответы — не галлюцинации, одобрены человеком.
- **Скорость:** нет GPT-вызова → мгновенный ответ.
- **Экономия:** меньше токенов на частые вопросы.
- **Continuous improvement:** система умнеет по мере накопления 👍.

**Связь с существующим:**
- `Message.feedback = up` → источник данных.
- `Message.ideal_answer` → эталонный текст.
- `/review` → интерфейс для одобрения и редактирования.

**Pipeline:**
```
Вопрос пользователя
    ↓
Semantic search в approved_answers (similarity > 0.85?)
    ↓ да                           ↓ нет
Instant answer               Обычный RAG pipeline
(без GPT)                         ↓
                            GPT ответ + возможность 👍
```

**Effort:** 3–4 дня (Уровень 1)
**Приоритет:** P1 — замыкает петлю обучения системы.
