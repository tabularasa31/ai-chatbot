# Словарь терминов для ручного тестирования Chat9

Краткие пояснения **простым языком** для тестировщика. Подробный план проверок: [`PRODUCT-QA-TEST-PLAN.md`](./PRODUCT-QA-TEST-PLAN.md).

---

| Термин | Простыми словами |
|--------|------------------|
| **Стенд** | Тестовая копия сайта (не обязательно «боевой» прод). У команды спроси точный **URL**. |
| **Smoke / дымовой тест** | Очень быстрая проверка: «сайт вообще живой?» Если smoke падает — дальше обычно не идут, пишут **блокер**. |
| **Блокер** | Такая поломка, что без неё нельзя нормально тестировать остальное (например, не открывается вход). |
| **Регрессия** | То, что **раньше работало**, а после изменений **сломалось**. |
| **Edge case** | Нестандартный, «редкий» сценарий (например, пользователь не подтвердил почту, но как-то попал внутрь). |
| **Валидация формы** | Сайт не даёт отправить форму с ошибками и показывает сообщение (пустое поле, неверный email и т.д.). |
| **Редирект** | Автоматический переход на другую страницу (например, после входа — на дашборд). |
| **403** | «Доступ запрещён» (часто видно в DevTools → Network рядом с запросом). |
| **Эмбеддинг** | Внутри продукта текст документа переводится в числа, чтобы искать **по смыслу**, а не только по точным словам. |
| **Чанк** | Кусок документа, по которому строится поиск. |
| **Rate limit** | Ограничение: «слишком много запросов за короткое время» — может появиться ошибка. |
| **RAG** | Схема работы бота: найти релевантные куски из твоих документов + сгенерировать ответ моделью. |
| **Client / workspace** | «Рабочее пространство» пользователя в системе: свои документы, ключи, настройки. |
| **Верификация email** | Процесс подтверждения почты: после регистрации на email приходит письмо, пользователь кликает по ссылке — и аккаунт активируется. До верификации доступ к функциям закрыт. |
| **Verification gate / гард** | Защитный экран, который показывается на **всех** страницах приложения, если email не подтверждён. Вместо ошибок API пользователь видит единый экран «Verify your email to continue». |
| **JWT / токен** | Короткоживущий «пропуск» в браузере (хранится в localStorage), который подтверждает личность пользователя при каждом запросе к серверу. Выдаётся сразу при регистрации — ещё до верификации email. Без токена любой защищённый URL редиректит на `/login`. |
| **localStorage** | Место в браузере, где сайт хранит данные (в том числе JWT-токен) между перезагрузками страницы. Очищается при явном выходе (Logout). |
| **Авторедирект** | Когда страница сама переходит на другую без клика пользователя. Например: кликнул по ссылке из письма верификации → страница `/verify` автоматически переходит на `/dashboard`, если токен уже есть в браузере. |

---

### Коды фич в документации (FI-032, FI-ESC и т.д.)

Это **внутренние метки** фич в репозитории. Для теста важнее **поведение на экране**, чем запоминание кода. Если в чеклисте указан код — открой связанный QA-файл или спроси лида.

---

*Словарь можно дополнять по мере появления новых терминов в чеклистах.*
