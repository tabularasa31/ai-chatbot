# Gap Analyzer — ручной QA-чеклист (RU, для джуна)

Этот документ помогает вручную проверить страницу **Gap Analyzer**. Он написан простым языком и подходит тестировщику, который только входит в проект.

---

## 1. Что это вообще такое

`Gap Analyzer` — это страница в дашборде Chat9, где клиент видит:

- какие темы в документации могут быть покрыты слабо;
- какие вопросы пользователи задают слишком часто;
- что можно скрыть, вернуть обратно или превратить в draft для новой статьи.

Тебе не нужно разбираться в ML или embeddings. Твоя задача проще:

- открыть страницу;
- проверить, что она не ломается;
- проверить, что действия работают предсказуемо;
- убедиться, что активный список и архив ведут себя логично.

---

## 2. Что подготовить перед тестом

Нужны:

- аккаунт с подтверждённым email;
- доступ в дашборд;
- желательно тестовый tenant, где уже есть документы и хотя бы несколько чатовых сессий;
- если данных мало, попроси у команды стенд или tenant с уже заполненным Gap Analyzer.

Почему это важно:

- на пустом tenant страница может честно показать пустое состояние;
- это не баг само по себе.

---

## 3. Smoke test: самое быстрое

- [ ] Войти в дашборд под верифицированным пользователем.
- [ ] В левом меню есть пункт **Gap Analyzer**.
- [ ] Переход на `/gap-analyzer` открывает страницу без белого экрана и без бесконечной загрузки.
- [ ] Видны заголовок, summary-карточки, блоки **Mode A** и **Mode B**.
- [ ] Кнопка **Recalculate now** отображается и нажимается.

Если уже тут всё ломается, это баг высокого приоритета.

---

## 4. Проверка доступа

- [ ] Пользователь без логина не может открыть `/gap-analyzer` напрямую.
- [ ] Неверифицированный пользователь не получает доступ к странице.
- [ ] Пользователь одного клиента не видит gap'ы другого клиента.

---

## 5. Active view

### Что проверить глазами

- [ ] По умолчанию открывается **Active view**.
- [ ] В summary есть карточки вроде `Active gaps`, `Uncovered`, `Partial`, `Last updated`.
- [ ] В левой колонке есть **Mode A**, в правой — **Mode B**.
- [ ] У карточек есть понятные бейджи статуса и coverage.

### Если есть данные

- [ ] Для Mode A и Mode B список отображается без визуальной поломки.
- [ ] Если элемент помечен `new`, бейдж виден аккуратно.
- [ ] Если элемент связан с docs-side проблемой, может быть бейдж `also missing in docs`.
- [ ] Текст в карточке не обрезается критично и не ломает верстку.

### Если данных нет

- [ ] Вместо крэша или пустого белого блока показывается внятное empty state.

---

## 6. Archive view

- [ ] Кнопка **Archive view** переключает страницу в архивный режим.
- [ ] В архиве summary меняется: появляются архивные счётчики, а не active summary.
- [ ] Для Mode A архив показывает dismissed docs-side темы.
- [ ] Для Mode B архив показывает отдельно `Closed` и отдельно `Dismissed`.
- [ ] Возврат в **Active view** работает без перезагрузки страницы руками.

Важно:

- если архив пустой, должен быть нормальный empty state, а не сломанный layout.

---

## 7. Фильтры

### Mode A filter

- [ ] У Mode A есть фильтр: `Active`, `Archived`, `Dismissed only`, `All`.
- [ ] При переключении фильтра список меняется.
- [ ] Нет ситуации, когда выбран один фильтр, а UI явно показывает данные от другого.

### Mode B filter

- [ ] У Mode B есть фильтр: `Active`, `Archived`, `Closed`, `Dismissed`, `All`.
- [ ] При переключении фильтра список меняется.
- [ ] В архивном режиме `Archived` работает предсказуемо и не смешивает всё в одну неразборчивую кучу.

---

## 8. Действие Recalculate

- [ ] Нажать **Recalculate now**.
- [ ] Кнопка на время становится disabled или показывает состояние запуска.
- [ ] Появляется понятное сообщение об успехе, например что recalculation accepted.
- [ ] Страница не падает после нажатия.

Что не требовать как обязательное:

- моментального появления новых gap'ов;
- мгновенного полного обновления данных.

Это orchestration-style действие, а не обещание мгновенного пересчёта прямо на глазах.

---

## 9. Действие Dismiss

Если есть активный элемент:

- [ ] Нажать **Dismiss** на карточке.
- [ ] Во время запроса кнопка не даёт спамить много раз подряд.
- [ ] После успеха появляется notice.
- [ ] Элемент уходит из активного списка или меняет состояние предсказуемо.
- [ ] В архиве этот элемент виден там, где ожидается по source/status.

Это один из самых важных сценариев.

---

## 10. Действие Reactivate

Если есть dismissed элемент в архиве:

- [ ] Нажать **Reactivate**.
- [ ] После успеха появляется notice.
- [ ] Элемент возвращается в активный список или хотя бы исчезает из архивного dismissed списка после reload.

---

## 11. Действие Draft

Если есть любой gap item:

- [ ] Нажать **Draft**.
- [ ] Открывается панель с draft preview.
- [ ] В панели видны title и markdown-текст.
- [ ] Кнопка **Copy** не ломает страницу.
- [ ] Кнопка **Close** закрывает панель.

Важно:

- draft — это preview, а не публикация.
- если API вернул ошибку, должна быть понятная ошибка, а не silent failure.

---

## 12. Что проверить для linked / dedupe поведения

Если на стенде есть связанные Mode A и Mode B элементы:

- [ ] В active view нет очевидного дубля одной и той же проблемы подряд в обеих колонках без причины.
- [ ] Если у Mode B карточки есть hint `also missing in docs`, он отображается корректно.
- [ ] Archive view остаётся source-specific и не прячет активные элементы просто потому, что у них есть связанная пара в архиве.

Если на стенде нет linked кейсов, это нужно отметить в отчёте как **не удалось проверить из-за отсутствия данных**, а не придумывать результат.

---

## 13. Негативные сценарии

- [ ] Если backend вернул ошибку, на странице есть понятный error banner.
- [ ] После ошибки пользователь всё ещё может повторить попытку.
- [ ] При пустом результате UI не ломается.
- [ ] При длинных label / example questions карточки остаются читаемыми.

---

## 14. Что приложить к багу

Если нашёл проблему, приложи:

1. шаги по порядку;
2. что ожидалось;
3. что получилось фактически;
4. скрин или короткое видео;
5. какой tenant / какой аккаунт / какой браузер;
6. если есть — текст ошибки из Network tab.

Хороший баг:

> Открыла `/gap-analyzer`, переключила `Archive view`, затем у Mode B выбрала `Archived`. Ожидала увидеть archived items или empty state. Фактически правая колонка зависла на skeleton и не восстановилась после 30 секунд.

---

## 15. Когда это точно стоит эскалировать сразу

Сразу пиши лидy, если:

- страница вообще не открывается;
- действия `Dismiss`, `Reactivate` или `Draft` стабильно ломают UI;
- видны данные другого клиента;
- архив и активный список ведут себя настолько странно, что теряются элементы или дублируются без причины.
