# FI-ESC — L2 Escalation Tickets: чеклист для тестировщика

**Фича:** когда бот не может уверенно ответить, пользователь просит человека или нажимает «Talk to support», в системе создаётся **тикет** (`ESC-0001`, …), владелец тенанта получает **email**, в дашборде есть раздел **Escalations**. Пользователь видит текст от бота (язык — по диалогу / подсказке локали), в конце ответа может быть скрытый маркер `[[escalation_ticket:ESC-####]]` (в UI виджета маркер убирается, номер показывается в баннере).

**Реестр фич:** [`docs/IMPLEMENTED_FEATURES.md`](../IMPLEMENTED_FEATURES.md)  
**Код / API (кратко):** `backend/escalation/`, `backend/chat/service.py`, `backend/routes/widget.py`, миграция `fi_esc_v1`.

---

## Перед тестом

1. Стенд с **применённой миграцией** `fi_esc_v1` (на пустой БД тикеты не создадутся).
2. У тестового клиента задан **OpenAI API key** (и на стенде настроен **Brevo**, если проверяете письма).
3. Два аккаунта **не обязательны** для базового сценария; для изоляции — два клиента / два пользователя: тикеты клиента A не видны клиенту B.

## A. Дашборд: раздел Escalations

| # | Действие | Ожидание |
|---|----------|----------|
| A1 | Войти под владельцем клиента, открыть **`/escalations`** | Страница открывается, в навбаре есть пункт **Escalations** |
| A2 | Фильтр **Status** (All / open / in_progress / resolved) | Список меняется или остаётся пустым без ошибки |
| A3 | Создать тикет (см. блок B или C), обновить список (**Refresh**) | Появилась строка: **Ticket #**, укороченный вопрос, **Priority**, **Status**, **User**, **Created** |
| A4 | Клик по строке — раскрытие | Видны **trigger**, полный **primary question**, при наличии — summary, user note, preview чанков |
| A5 | Для тикета со статусом не **resolved**: ввести текст в **Mark resolved** → **Mark as resolved** | Статус становится **resolved**, появляется **Resolution**; повторное разрешение не требуется |

## B. Виджет: автоматическая эскалация (нет знаний / низкая уверенность)

| # | Действие | Ожидание |
|---|----------|----------|
| B1 | Клиент **без документов** или вопрос **вне базы** | Ответ содержит fallback «недостаточно информации…» и **дополнительный абзац** про передачу в поддержку / тикет |
| B2 | Проверить баннер над чатом | Отображается **Support ticket: ESC-…** (если в ответе был маркер тикета) |
| B3 | В списке сообщений | Текст **без** строки `[[escalation_ticket:…]]` (маркер не показывается как обычный текст) |
| B4 | **`chat_ended`** | После обычного эскейта по RAG чат **не** закрыт: поле в JSON ответа виджета `chat_ended: false` (DevTools → Network на прокси `/widget/chat`) |

## C. Виджет: явный запрос человека (до RAG)

| # | Действие | Ожидание |
|---|----------|----------|
| C1 | Написать, например: *«I want to talk to a human»* или *«хочу поговорить с человеком»* | Ответ про передачу в поддержку / запрос email (если email не известен из KYC), тикет в дашборде |
| C2 | Если включён **KYC** и в контексте уже есть **email** | Реже спрашивают email; возможен сразу handoff + follow-up (да/нет) — зафиксировать фактическое поведение |

---

## D. Виджет: кнопка **Talk to support**

| # | Действие | Ожидание |
|---|----------|----------|
| D1 | Отправить хотя бы одно сообщение (чтобы появился `session_id`) | Кнопка **Talk to support** активна |
| D2 | Без сообщений (пустая сессия) | Кнопка неактивна или запрос даёт ошибку — зафиксировать по продукту |
| D3 | Нажать **Talk to support** | Появляется ответ ассистента **без** лишнего пользовательского «фейкового» сообщения в истории; тикет создаётся |
| D4 | Баннер тикета | Номер **ESC-####** совпадает с записью в `/escalations` |

## E. Сбор email и follow-up (если тикет без email)

| # | Действие | Ожидание |
|---|----------|----------|
| E1 | После эскейна бот просит email — отправить **валидный** email одной строкой | Подтверждение, снятие режима «ждём email», дальше вопрос «нужна ли ещё помощь» (формулировка на языке пользователя) |
| E2 | Отправить **мусор** вместо email | Просьба ввести корректный email (без падения UI) |
| E3 | На вопрос «нужна ли ещё помощь» ответить **нет** / отказ | Чат переходит в **закрытое** состояние: следующий запрос — короткий ответ «чат закрыт»; в JSON **`chat_ended: true`** |
| E4 | После закрытия | Поле ввода и **Send** заблокированы; **Talk to support** не должен ломать страницу |

## F. Локаль (виджет)

| # | Действие | Ожидание |
|---|----------|----------|

## G. API `POST /chat` (опционально, Postman)

| # | Действие | Ожидание |
|---|----------|----------|
| G1 | `POST /chat` с **X-API-Key**, телом `{ "question": "..." }` | В ответе есть **`chat_ended`** (boolean) |
| G2 | Заголовок **`X-Browser-Locale: de-DE`** | Запрос проходит, локаль подмешивается в контекст (косвенно — через поведение эскалации) |
| G3 | `POST /chat/{session_id}/escalate` с тем же ключом, телом `{ "trigger": "user_request", "user_note": null }` | `200`, поля `message`, `ticket_number`; в истории чата новое сообщение **assistant** |

## H. Изоляция и безопасность

| # | Действие | Ожидание |
|---|----------|----------|
| H1 | JWT `GET /escalations` пользователя A | Только тикеты клиента A; чужие `ticket_id` → **404** на `GET`/`POST .../resolve` |
| H2 | Публичный **`POST /widget/escalate`** с чужим `session_id` | **404** Session not found (не создавать тикет на чужой чат) |

## Регрессия (коротко)

- [ ] Обычный RAG-диалог **с документами** по-прежнему даёт ответы без лишних тикетов при высокой релевантности.
- [ ] Лимиты: **`/widget/chat`** и **`/widget/escalate`** — не ниже заявленного в продукте (например 20/мин с одного IP).
- [ ] **Логи / сессии** в дашборде: сообщения эскалации видны как обычные реплики ассистента.
