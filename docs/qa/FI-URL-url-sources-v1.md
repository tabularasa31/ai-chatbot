# FI-URL: URL Sources v1 (парсинг сайта документации)

## Что это за фича

Теперь в **Knowledge** можно добавить не только файл, но и **URL** сайта документации. Система:

- принимает корневой URL;
- делает preflight-проверку;
- находит страницы этого же домена (с лимитом);
- извлекает читаемый текст;
- создаёт документы типа `url` и эмбеддинги;
- показывает статус, прогресс, ошибки и историю запусков в интерфейсе.

Это **v1**, поэтому есть сознательные ограничения:

- только `http/https`;
- только **один домен** на source;
- лимит до **50 страниц** на источник;
- поддерживаются расписания только `daily`, `weekly`, `manual`;
- можно исключать пути через `exclusions`;
- если у клиента нет OpenAI key, источник создаётся, но индексирование ставится на паузу.

## Что важно проверить

### Подготовка

- [ ] Есть пользователь с **подтверждённым email**.
- [ ] Создан client / workspace.
- [ ] Для основного happy path у клиента задан **рабочий OpenAI API key**.
- [ ] Есть публичный сайт документации или небольшой тестовый сайт на одном домене.

### Основной happy path

- [ ] Открыть страницу **Knowledge**.
- [ ] Найти карточку / форму для добавления **URL source**.
- [ ] Ввести валидный URL вида `https://docs.example.com`.
- [ ] Выбрать допустимое расписание (`daily`, `weekly` или `manual`).
- [ ] Сохранить источник.
- [ ] Новый источник появляется в списке без перезагрузки страницы или после обычного refresh.
- [ ] Сразу после создания виден ожидаемый начальный статус (`queued`, `indexing` или `paused` — зависит от наличия OpenAI key).
- [ ] После завершения индексации источник доходит до `ready`, если сайт доступен и страницы читаемые.

### Что должно быть видно в UI

- [ ] У источника видны: имя, URL, статус, расписание, число найденных / проиндексированных страниц, дата последнего запуска.
- [ ] Если достигнут лимит страниц, в UI есть предупреждение про лимит, а интерфейс не ломается.
- [ ] Если есть ошибки отдельных страниц, вся страница Knowledge остаётся рабочей.
- [ ] Детальная страница source открывается и показывает:
  - последние запуски;
  - список проиндексированных страниц;
  - число чанков по страницам.
- [ ] В detail-экране история запусков показывает **только последние 5 запусков**, новые сверху.

### Refresh / update / delete

- [ ] Кнопка **Refresh** запускает повторный crawl.
- [ ] Повторный refresh слишком рано даёт понятную ошибку / ограничение, а не бесконечную загрузку.
- [ ] Можно изменить имя, расписание и exclusions; после сохранения изменения остаются после обновления страницы.
- [ ] Можно удалить URL source; после удаления он исчезает из списка, связанные URL-документы больше не видны.

### Проверка валидации

- [ ] `ftp://...` или URL без схемы не принимается.
- [ ] Значение расписания вне списка (`monthly`, случайный текст) не принимается.
- [ ] Некорректный URL показывает понятную ошибку.
- [ ] URL с логином/паролем в адресе не должен приниматься.

### Безопасность / SSRF

- [ ] `http://127.0.0.1/...` не принимается.
- [ ] `http://localhost/...` не принимается.
- [ ] URL, который редиректит на `localhost` или приватный IP, не должен индексироваться.
- [ ] Интерфейс в этих случаях показывает понятную ошибку, а не зависает.

### Ошибки внешнего сайта

- [ ] Если сайт возвращает `404`, пользователь получает понятное сообщение, что страница не найдена.
- [ ] Если сайт требует доступ (`401/403`), пользователь получает понятное сообщение, что URL недоступен для индексации.
- [ ] Если сайт временно недоступен (`5xx`), система показывает понятную временную ошибку.
- [ ] Если ответ слишком большой, источник не валит страницу и показывает контролируемую ошибку.

### Поведение без OpenAI key

- [ ] Если создать URL source без OpenAI key у клиента, источник создаётся в списке, но статус переходит в `paused`.
- [ ] В UI видно понятное сообщение, что индексирование остановлено из-за отсутствия ключа.
- [ ] После добавления OpenAI key можно запустить refresh и довести источник до `ready`.

### Изоляция и права доступа

- [ ] Пользователь другого клиента не видит чужие URL sources в списке.
- [ ] Пользователь другого клиента не может открыть detail чужого source по прямой ссылке / ID.
- [ ] Создание, редактирование, refresh и удаление доступны только подтверждённому пользователю.

### Регрессия

- [ ] Обычная загрузка файлов в Knowledge продолжает работать как раньше.
- [ ] Документы-файлы и URL sources одновременно отображаются корректно.
- [ ] Чат продолжает отвечать по уже проиндексированным файлам и URL-страницам.
- [ ] Страница Knowledge не ломается, если у пользователя ещё нет client: ожидаемое поведение — понятная ошибка, а не пустой сломанный экран.

## Полезные подсказки для QA

- Если источник не индексируется, полезно открыть DevTools → **Network** и посмотреть ответ API.
- Если не ясно, это ошибка UI или backend, зафиксируй:
  - какой URL добавляли;
  - был ли OpenAI key;
  - какой статус показал UI;
  - текст ошибки;
  - появился ли source в списке после refresh страницы.

## Что приложить к багу

- URL, который тестировали.
- Скрин списка source или detail-экрана.
- Шаги воспроизведения.
- Ожидаемое поведение.
- Фактическое поведение.
- Ответ API / статус из Network, если есть.
