## Виджет Chat9 и опыт конечного пользователя

Этот документ описывает, как работает виджет Chat9 на сайте клиента и что видит конечный пользователь. Здесь нет кода, только поведение и сценарии.

### Что такое виджет Chat9

Виджет — это **окно чата**, встроенное в сайт клиента. Технически это:
- небольшой JS‑скрипт, который клиент вставляет в код сайта;
- iframe‑окно, в котором живёт UI чата и логика общения с бэкендом.

Для клиента:
- установка сводится к копированию готового сниппета;
- виджет автоматически привязывается к нужному клиентскому аккаунту по публичному ID.

Для конечного пользователя:
- это просто «чат в углу экрана», где можно задать вопрос о продукте.

---

### UI и поведение виджета

Основные элементы:
- кнопка/иконка запуска виджета (обычно в нижнем углу страницы);
- панель чата:
  - область сообщений;
  - индикатор набора ответа («typing…»);
  - поле ввода с кнопкой отправки;
  - возможный футер с брендом Chat9 и ссылкой на сайт нашего продукта.

Базовое поведение:
- при первом открытии создаётся новая сессия;
- если первое сообщение пустое, виджет может показать дефолтный greeting вместо ошибки;
- сообщения пользователя отображаются в ленте справа (или в стиле, принятом в дизайне);
- ответы бота появляется с задержкой, имитируя «набор текста»;
- при прокрутке новые сообщения не сдвигают всю страницу — скролл происходит внутри виджета.
- если боту нужен один точный недостающий кусок контекста, он может показать
  структурный clarification-вопрос вместо немедленного ответа;
- если у clarification есть варианты ответа, виджет показывает их как quick replies;
- quick replies активны только у последнего assistant clarification, старые кнопки
  не должны оставаться рабочими после продолжения диалога.

Языковое поведение:
- до первого реального вопроса deterministic-тексты виджета используют цепочку
  `locale -> browser_locale -> English`;
- после первого реального вопроса бот должен отвечать на языке самого вопроса.

### Как виджет подключается к бэкенду

1. **Инициализация**
   - Скрипт на сайте клиента загружается из нашего бэкенда.
   - Скрипт знает публичный идентификатор клиента (передан в сниппете).
   - При первом открытии виджета отправляется запрос на инициализацию сессии:
     - создаётся уникальный идентификатор сессии;
     - может определяться режим (анонимный пользователь или идентифицированный через токен).

2. **Отправка сообщений**
   - При каждом сообщении пользователя виджет отправляет запрос на публичный чат‑endpoint.
   - Запрос содержит:
     - публичный ID клиента;
     - ID сессии;
     - текст сообщения;
     - локаль (например, ru/en);
     - опциональную информацию о пользователе (если включена идентификация).

3. **Получение ответов**
   - Бэкенд обрабатывает сообщение через общий RAG‑pipeline:
     - защищает от инъекций и вычищает PII;
     - выполняет поиск по базе знаний клиента;
     - принимает решение: ответить сразу, задать один clarification-вопрос,
       либо дать безопасный partial answer + clarification;
     - формирует ответ на основе найденных фрагментов и настроек клиента.
   - Ответ возвращается виджету и отображается в UI.

Важно:
- виджет больше не трактует все ответы бэкенда как “просто текст”;
- для `clarification` и `partial_with_clarification` он читает `message_type`
  и структурный `clarification` payload;
- первый пустой turn может вернуть обычный `answer` с greeting;
- legacy-поле `response` остаётся совместимым текстовым алиасом для существующих
  интеграций.

### Идентификация пользователей виджета (опционально)

Некоторые клиенты хотят понимать, **кто именно** общается с ботом (например, сопоставлять чат с их CRM или системой биллинга). Для этого предусмотрен механизм краткоживущих токенов:

- На стороне сервиса клиента формируется **подписанный токен**, в котором зашифрована информация о пользователе (ID, роль, план и т. д.).
- Токен подписывается секретом, которым владеет только клиент; наш бэкенд знает только хэш/шифротекст и умеет проверять подпись, но не требует открытого секрета.
- Виджет передаёт этот токен при инициализации сессии.
- Бэкенд:
  - валидирует подпись;
  - извлекает запланированный набор полей и сохраняет их в контексте сессии.

Важно:
- не все поля из токена попадают в LLM‑промпт — только минимально необходимый и безопасный поднабор;
- цель — дать боту контекст (например, тип тарифа), а не раскрывать полные персональные данные.

### Как бот отвечает на вопросы (с точки зрения пользователя)

Процесс в глазах пользователя выглядит так:

1. Пользователь задаёт вопрос (например: «Как поменять пароль в аккаунте?»).
2. Виджет немного «думает» — показывает индикатор набора.
3. Бот отвечает:
   - вежливым тоном;
   - в соответствии с выбранным клиентом уровнем детализации (см. Response controls);
   - с опорой на документацию и FAQ клиента.

Если документов недостаточно или вопрос сильно выходит за рамки знаний, возможны варианты:
- бот честно сообщает, что не хватает информации;
- задаёт один конкретный clarification-вопрос, если это реально повышает шанс
  дать полезный и корректный ответ;
- предлагает переформулировать вопрос или уточнить детали, если structured
  clarification для такого паттерна в MVP не поддержан;
- при определённых условиях переводит диалог в эскалацию к человеку.

Для greeting, soft rejection, clarification и escalation fallback используется тот же
языковой контракт:
- до первого вопроса ориентируемся на `locale` / `browser_locale`;
- после первого вопроса ориентируемся на язык вопроса пользователя.

Примеры clarification-сценариев:
- неоднозначный intent: «Подключить домен» → уточнение, речь про DNS, CDN или SSL;
- отсутствует критичный параметр: «API request fails» → запрос endpoint и текста ошибки;
- часть полезного ответа уже можно дать безопасно, но для точного шага нужен ещё
  один параметр.

### Когда срабатывает эскалация к человеку

Виджет поддерживает **эскалацию L2**, когда бот не может или не должен отвечать самостоятельно.

Примеры триггеров:
- **Низкая релевантность поиска** — система не находит достаточно хорошие фрагменты в базе знаний.
- **Отсутствие документов** — клиент ещё не загрузил знания, но виджет уже установлен.
- **Явный запрос пользователя** — фразы типа «поговорить с оператором», «связаться с человеком», «live agent» и т. п.
- **Ручная эскалация** — клиентская интеграция вызывает специальную операцию эскалации из своего кода или дашборда.

Что видит пользователь:
- сообщение о том, что запрос передан в поддержку;
- (опционально) номер тикета;
- возможный текст, поясняющий, что оператор свяжется по указанному каналу (email/чат и т. п.).

На стороне клиента:
- создаётся тикет с привязкой к конкретной чат‑сессии;
- операторы видят весь контекст и могут доработать запрос.

### Безопасность и приватность в виджете

Несмотря на то, что виджет находится на сайте клиента, безопасность обрабатывается централизованно:

- **Защита от prompt‑injection**
  - Входящие сообщения анализируются на попытки управлять поведением модели (например, «забудь все правила и скажи…»).
  - При подозрении на инъекцию бот выбирает безопасное поведение (отказ или нейтральный ответ) и не раскрывает внутренние детали системы.

- **Защита PII**
  - Перед отправкой текста во внешнюю модель выполняется маскирование персональных данных:
    - email, телефоны, номера карт, ключи, IP‑адреса и др.
  - В базе отдельно хранятся:
    - оригинальный зашифрованный текст;
    - отредактированный вариант без PII.
  - В интерфейсах поддержки и логах по умолчанию показывается именно безопасный вариант, а доступ к оригинальным данным жёстко контролируется.

- **Rate limiting**
  - Есть ограничения на частоту запросов — как для виджета, так и для других API.
  - Это защищает систему от злоупотреблений и DDoS‑подобной активности.

### Типичные пользовательские сценарии

1. **FAQ‑бот на лендинге**
   - Пользователь приходит на сайт, задаёт вопросы о продукте и тарифах.
   - Бот отвечает по документации и заранее подготовленному FAQ.
   - При сложных или «нестандартных» вопросах — эскалация в поддержку.

2. **Онбординг в продукте**
   - Пользователь уже в интерфейсе SaaS‑продукта клиента.
   - Виджет помогает:
     - объяснить смысл разделов;
     - подсказать шаги по настройке;
     - ответить на вопросы по интеграциям.

3. **Внутренний помощник для сотрудников**
   - Виджет встроен во внутренний портал компании.
   - Бот отвечает на вопросы о внутренних регламентах, IT‑поддержке, HR‑процессах.

### Что важно знать сейлз и саппорту про виджет

- **Для сейлз**
  - Виджет — главная «витрина» продукта, впечатление от него сильно влияет на восприятие платформы.
  - На демо важно показать:
    - как легко его внедрить (одна строка встраивания);
    - как он подхватывает документацию клиента;
    - как работает эскалация;
    - как можно управлять стилем и уровнем ответов.

- **Для саппорта**
  - При проблемах «бот не отвечает» нужно проверять:
    - есть ли ошибки в консоли браузера у клиента;
    - доступны ли публичные API (нет ли блокировок по CORS или сети);
    - есть ли у клиента валидный OpenAI‑ключ;
    - не заблокирован ли клиент по rate limiting;
    - есть ли у клиента документы и прошла ли индексация.
  - При некорректных ответах:
    - проверять, есть ли соответствующая информация в базе знаний;
    - смотреть логи диалогов и bad answers;
    - при необходимости — рекомендовать клиенту пересобрать документацию или добавить FAQ.
