# Chat9 — план ручного тестирования (для начинающего тестировщика)

Этот документ написан **для джуна**: здесь объясняется, **что за продукт**, **как вообще тестировать**, и **что проверять по шагам**. Не нужно знать программирование или архитектуру — достаточно внимательности и браузера.

---

## Для кого этот документ и что от тебя ждут

**Ты проверяешь продукт глазами пользователя:** кликаешь по кнопкам, заполняешь формы, смотришь, что написано на экране. Если что-то не так — **оформляешь баг** (описание шагов + ожидание + факт + скрин).

**Что мы тестируем в первую очередь**

- **Веб-интерфейс** (страницы в браузере).
- **Виджет** (чат, который встраивают на сайт).

**Что можно не трогать на старте**

- Ручные запросы в **Postman** к API — по желанию и по договорённости с командой.

**Полезная ссылка для сверки «что уже сделано в продукте»:** [`docs/IMPLEMENTED_FEATURES.md`](../IMPLEMENTED_FEATURES.md) (на английском, по модулям).

**Не знаешь термин из чеклиста?** Открой отдельный файл: **[`docs/qa/QA-GLOSSARY.md`](./QA-GLOSSARY.md)** (стенд, smoke, блокер, RAG, эмбеддинг и т.д.).

## Часть 1. Что такое Chat9 (очень кратко)

**Chat9** — сервис для компаний: загружают документацию (PDF, Markdown, Swagger и т.д.), система строит по ней базу знаний, пользователи задают вопросы **обычным языком** и получают ответы с опорой на эти материалы.

**Два типа людей**

1. **Владелец аккаунта** — регистрируется, подтверждает email, настраивает пространство и ключ **OpenAI** (без него часть функций не заработает).
2. **Посетители сайта клиента** — общаются через **виджет** чата, им не нужен логин в Chat9.

**Важно для тестов: изоляция**

У каждого аккаунта свои документы и чаты. **Нельзя**, чтобы один пользователь видел данные другого. Это проверяют **двумя разными email** (например `you+qa1@gmail.com` и `you+qa2@gmail.com`).

### Где что лежит в интерфейсе

| Где в интерфейсе | Зачем смотреть |
|------------------|----------------|
| Главная / лендинг | Описание, вход, регистрация |
| Регистрация, вход, «забыли пароль», подтверждение email | Доступ в продукт |
| Дашборд | API-ключ, OpenAI, код виджета, настройки ответа, Widget API |
| Knowledge | Загрузка файлов, URL sources, статусы, «здоровье» документа (FI-032) |
| Чат в приложении | Ответы по своим документам |
| Логи / сессии | История диалогов |
| Отзывы 👍👎, Review | Оценка и разбор ответов |
| Debug | Техническая страница (режим поиска, фрагменты) |
| Escalations | Тикеты «нужен человек» (подробно: `docs/qa/FI-ESC-escalation-tickets-qa.md`) |
| Админка | Только для админов — метрики и списки |
| Виджет + скрипт вставки | Публичный чат на сайте клиента |

### От чего продукт зависит снаружи

- **OpenAI** — без рабочего ключа у клиента не заработают эмбеддинги, чат, проверка здоровья документов.
- **Почта (например Brevo)** — письма подтверждения и сброса пароля. Не пришло — проверь **спам** и спроси команду про настройки стенда.
- **Сервер и база** — если API лежит, страницы покажут ошибки загрузки.

## Часть 2. Как готовиться и как писать баги

### Перед первым днём тестирования

1. Спроси у команды **ссылку на стенд** (и нужен ли отдельный **админ-аккаунт**).
2. Заведи **два тестовых email** (удобно плюс-алиасы: `имя+qa1@почта`, `имя+qa2@почта`).
3. Если разрешено — попроси **тестовый ключ OpenAI** (или узнай, есть ли общий на стенде).
4. Подготовь **маленькие файлы**: `.md` с заголовками, небольшой `.pdf` с текстом; при необходимости — маленький Swagger `.json` по примеру из документации продукта.

### Как оформить баг (минимум)

1. **Шаги** — нумерованный список: что нажал, что ввёл.
2. **Ожидалось** — как должно быть (по здравому смыслу или по подсказке лида).
3. **Фактически** — что увидел.
4. **Скрин** или короткое видео; **время**, **браузер**, **аккаунт** (email без пароля).

**Если что-то «красное» в интерфейсе или не грузится**

- Открой **DevTools** (обычно F12) → вкладка **Network**.
- Найди запрос с ошибкой, посмотри **код** (например 403, 500) и **текст ответа** — это полезно приложить к багу.

## Часть 3. Чеклисты по блокам

**Как работать с чеклистом**

- Иди **сверху вниз** (сначала вход и дашборд, потом документы и чат).
- Галочка `[ ]` — пункт ещё не проверен; после проверки можно заменить на `[x]` или отметить в твоей системе задач.
- Если не понятно **что считать успехом** — спроси лида или зафиксируй «как сейчас работает» и передай на согласование.

---

### Блок A. Дымовой тест — примерно 15–20 минут

**Цель:** убедиться, что продукт в целом открывается и основной путь не сломан.

- [ ] Открывается главная страница; в консоли браузера (F12 → Console) нет **критичных** красных ошибок (мелочь можно отметить отдельно).
- [ ] Получается **зарегистрироваться** новым пользователем **или** войти существующим (как договоритесь со стендом).
- [ ] После входа открывается **дашборд** (или другое главное место приложения — уточни у команды, если отличается).
- [ ] Открывается раздел **Knowledge**.
- [ ] **Выход** из аккаунта и **повторный вход** работают.

**Если что-то из этого не работает** — это часто **блокер**: остальное можно отложить и сначала описать эту проблему.

---

### Блок B. Аккаунт, почта, пароль, доступ без входа

**Сначала простые проверки**

- [ ] **Регистрация:** пустые поля и неверный формат email — сайт **не** отправляет форму молча, показывает понятное сообщение.
- [ ] **Без входа:** если вставить в адресную строку `/dashboard` (и другие разделы приложения) — либо **редирект на логин**, либо честный отказ в доступе (не «белый экран без объяснений»).
- [ ] **Вход:** неверный пароль — понятная ошибка; верный пароль — успешный вход.

#### B1. Подтверждение email (верификация)

- [ ] **После регистрации:** появляется экран **«Check your inbox»** с **твоим** email; кнопки «Go to sign in» нет. Есть кнопка **«← Back»** — нажатие возвращает на форму регистрации (например, если ошибся в email).
- [ ] **Без верификации попасть в дашборд нельзя (прямой URL):** вставь `/dashboard` или `/logs` в адресную строку — редирект на `/login`. Токена нет, доступа нет.
- [ ] **Попытка войти без верификации (через форму логина):** введи email и пароль зарегистрированного, но **не верифицированного** аккаунта → появляется ошибка **«Email not verified. Please check your inbox.»** прямо в форме; токен **не** выдаётся; в дашборд не попасть.
- [ ] **Письмо:** на почту приходит письмо со ссылкой; ссылка ведёт на страницу вида `/verify?token=…`.
- [ ] **По ссылке из письма:** страница `/verify` отображает «Confirming your link…», затем **сразу** переходит на `/dashboard`; дашборд открывается нормально с API-ключом и кодом виджета.
- [ ] **По ссылке из письма в другом браузере:** поведение то же — верификация проходит, токен сохраняется, редирект на `/dashboard`.
- [ ] **Старая или уже использованная ссылка подтверждения:** страница `/verify` показывает «Verification failed» с понятным текстом ошибки и кнопкой «Go to Sign in»; приложение не «ломается».
- [ ] **Повторный клик по уже использованной ссылке:** то же поведение — ошибка с предложением войти.

#### B2. Восстановление пароля

- [ ] **Обычный сценарий (верифицированный пользователь):** «Забыли пароль» → письмо → ссылка → новый пароль → вход с новым паролем — **успех**.
- [ ] **Сброс пароля для неверифицированного пользователя:** зарегистрировался, **не** кликал по письму верификации → «Forgot password» → письмо → ссылка сброса → задал новый пароль → вошёл → дашборд **открывается полностью** без экрана «Verify your email» (сброс через ссылку из почты = подтверждение владения email, это ожидаемое поведение).
- [ ] **Forgot password для неверифицированного email:** ответ системы **одинаковый** с обычным случаем — «If this email is registered, you'll receive a password reset link shortly» (система не раскрывает статус аккаунта).
- [ ] **Ссылка сброса старше ~1 часа:** страница `/reset-password` сообщает, что ссылка недействительна; предлагает запросить новую.
- [ ] **Два поля пароля не совпадают:** до отправки на сервер показывается ошибка «Passwords do not match».
- [ ] **Forgot password для несуществующего email:** ответ **одинаковый** с обычным случаем (система **не** говорит «такого пользователя нет»).
- [ ] **После успешного сброса пароля:** страница `/reset-password` показывает «Password updated!» и через ~2 секунды редиректит на `/login`; вход с новым паролем работает.

---

### Блок C. Дашборд, клиент, ключи, настройки

- [ ] При первом нормальном входе после подтверждения создаётся клиент / видны **API key** и при необходимости **public_id** для виджета.
- [ ] **OpenAI API key:** пустой и явно неверный формат (не похож на `sk-…`) — ожидаемое поведение; **валидный** (если есть) — после сохранения интерфейс показывает, что ключ задан.
- [ ] **Имя клиента / профиль** (если есть в UI) — после сохранения и **обновления страницы** значение на месте.
- [ ] **Код встраивания виджета** виден, кнопка «Копировать» не ломает страницу.
- [ ] **Response controls** (страница `/settings/disclosure`, настройка «как подробно отвечать»): открывается, есть уровни (Detailed / Standard / Corporate), **Save** сохраняет выбор после перезагрузки. Без подтверждённой почты защищённые действия ведут себя предсказуемо (как остальные настройки).
- [ ] **Widget API** (`/settings/widget`): при отсутствии секрета — можно сгенерировать; при наличии — маскированная подсказка; блок со статистикой за 7 дней не валит страницу.

---

### Блок D. Knowledge: files + URL sources

- [ ] Загружаются разрешённые типы: `.pdf`, `.md`, swagger `.json` / `.yaml` (сверься с подсказкой на экране).
- [ ] Неподдерживаемый тип и **слишком большой** файл — **понятное** сообщение, а не тишина.
- [ ] Если в продукте есть **лимит числа документов** — при достижении лимита новая загрузка блокируется с ясным текстом.
- [ ] В списке видны имя, тип, статус (**processing / ready / error**).
- [ ] В успешном сценарии документ доходит до **ready**; эмбеддинги запускаются так, как сейчас задумано в UI.
- [ ] **Удаление** документа — его нет в списке; тот же файл можно загрузить снова.
- [ ] Можно добавить **URL source**; источник появляется в списке и показывает статус краулинга / индексации.
- [ ] В основной строке URL source видны расписание, индексированные страницы, здоровье / warning и actions.
- [ ] Detail URL source открывается прямо под строкой и показывает последние запуски и exclusions.
- [ ] В detail URL source можно удалить одну проиндексированную страницу; она исчезает сразу и не возвращается после refresh этого source.
- [ ] Общий лимит knowledge считается суммарно для файлов и URL-страниц; при достижении лимита UI показывает понятное сообщение.
- [ ] Если URL source невалидный, ведёт на localhost / приватный IP или редиректит туда, UI показывает понятную ошибку.
- [ ] Если у клиента нет OpenAI key, URL source создаётся или отображается предсказуемо, без падения страницы; статус и ошибка понятны пользователю.

Подробный чеклист по URL sources:

- **`docs/qa/FI-URL-url-sources-v1.md`**

### Блок E. «Здоровье» документа (FI-032)

Подробный чеклист: **`docs/qa/FI-032-document-health-check.md`**.

Кратко:

- [ ] После появления эмбеддингов индикатор здоровья **не** крутится вечно.
- [ ] Цвета и подписи соответствуют уровням (Good / Fair / Needs attention).
- [ ] Предупреждения видны при наведении / в раскрывающемся списке.
- [ ] Кнопка **Re-check** обновляет результат.
- [ ] Без ключа OpenAI — предсказуемая ошибка, **интерфейс не падает**.

### Блок F. Чат внутри приложения

- [ ] **PII (FI-043):** в **логах / истории** в дашборде виден **исходный** текст вопроса (если ввели email, телефон и т.д.). На сторону OpenAI для поиска и ответа уходит **замаскированная** версия (`[EMAIL]`, `[PHONE]` и т.п.). Сложные проверки «утечки в провайдера» — по договорённости с лидом (логи, прокси).
- [ ] **Уровень ответа (FI-DISC):** в Response controls выставить **Corporate**, задать технический вопрос по документу — ответ **сдержаннее / менее техничный**, чем при **Detailed** (субъективно; для регрессии сравни с **Standard**).
- [ ] С **готовыми эмбеддингами** задать вопрос по содержимому загруженного файла — ответ **про то, что в документе**, а не общие фразы.
- [ ] Вопрос **не по теме документа** — бот ограничивает ответ или честно говорит, что данных нет (зафиксируй фактическое поведение и согласуй с продуктом, если спорно).
- [ ] Пустой или слишком длинный вопрос — валидация или понятная ошибка.
- [ ] **Новая сессия** vs продолжение диалога — как в текущем UI и ТЗ.

---

### Блок G. Логи, лайки, разбор ответов

- [ ] В разделе логов видны прошлые диалоги, открывается история сообщений.
- [ ] Лайк / дизлайк сохраняется; после обновления страницы состояние **не** сбрасывается неправильно.
- [ ] Раздел плохих ответов / review (если есть): туда попадают оценённые сообщения; можно ли добавить заметку / «идеальный ответ» — как в UI.

---

### Блок H. Debug

*Часто для внутренних ролей; твоя задача — убедиться, что страница не ломается.*

- [ ] Страница доступна тем, кому положено (или всем — **уточни у команды**).
- [ ] Запрос выполняется, показываются найденные фрагменты и оценки совпадения; страница не падает и не сыпет ошибками в консоль.

---

### Блок I. Админка (только админы)

- [ ] Пользователь **без** прав админа **не** видит пункт меню и/или получает отказ (например 403) при прямой ссылке.
- [ ] Админ видит **метрики** и список клиентов; нет «NaN», страница не пустая при наличии данных.
- [ ] Лишние персональные данные чужих пользователей не светятся сверх того, что задумано (сверь с политикой, если есть).

---

### Блок J. Виджет и публичный сценарий

- [ ] Страница виджета открывается; сообщение уходит и приходит ответ при корректных **client id / public id** на стенде.
- [ ] Код с дашборда вставлен в тестовую HTML-страницу — виджет грузится, на разрешённом домене не ломается из-за CORS/безопасности.
- [ ] **FI-KYC (опционально, через API):** `POST /widget/session/init` без identity-токена → режим anonymous; с валидным токеном → identified; с подделанным — снова anonymous, виджет не падает.
- [ ] **Много сообщений подряд** (например 20+ в минуту): срабатывает ограничение — зафиксируй текст ошибки (**не** души прод настоящей атакой).
- [ ] **FI-ESC на виджете:** кнопка **Talk to support** после начала диалога создаёт тикет, баннер **Support ticket**; при закрытии чата поле ввода блокируется; в ответе API есть признак **`chat_ended`** (детали — в `FI-ESC-escalation-tickets-qa.md`).

---

### Блок M. Эскалации в дашборде (FI-ESC)

Детальный чеклист: **`docs/qa/FI-ESC-escalation-tickets-qa.md`**.

Кратко:

- [ ] `/escalations` открывается: фильтр, раскрытие строки, **Mark as resolved**.
- [ ] После создания тикета с виджета или из чата запись появляется в списке; письмо владельцу — если на стенде настроена почта.
- [ ] Второй аккаунт **не** видит чужие тикеты.

---

### Блок K. Браузеры и узкое окно

- [ ] Хотя бы **два** браузера из Chrome / Safari / Firefox: вход, загрузка документа, один диалог в чате.
- [ ] Узкое окно (как телефон): меню доступно, кнопки нажимаются, нет сильного горизонтального «слома» верстки на главных экранах.

---

### Блок L. Негативные сценарии (когда будет время)

- [ ] Обрыв сети **во время** загрузки файла — ошибка, можно повторить.
- [ ] Долго не трогал вкладку — при следующем действии просит **войти снова** (если сессия истекла).
- [ ] Два разных аккаунта в **двух окнах инкогнито** — документы и чаты **не** смешиваются.

---

## Часть 4. С чего начать в первый раз (порядок)

1. **Блок A** — дымовой тест.  
2. **B → C → D** — аккаунт и почта, дашборд, документы.  
3. **E** — здоровье документа (по файлу FI-032).  
4. **F → G** — чат, логи, отзывы.  
5. **J** — виджет.  
6. **M** — эскалации (по `FI-ESC-escalation-tickets-qa.md`).  
7. **I, H** — если дали доступ.  
8. **K, L** — по оставшемуся времени.
